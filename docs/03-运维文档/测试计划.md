## 2 测试策略

### 2.1 测试类型与工具

本项目采用多层次、多维度的测试策略，覆盖从代码级别到系统级别的完整测试链路。

#### 2.1.1 单元测试

- **工具**：JUnit 5 + Mockito + Spring Boot Test
- **范围**：各微服务的Service层业务逻辑、工具类方法、数据转换组件
- **覆盖率目标**：核心业务逻辑代码覆盖率不低于80%
- **执行方式**：开发人员在编码阶段同步编写，通过Maven/Gradle集成到CI流水线自动执行

#### 2.1.2 接口测试

- **工具**：Postman + Newman（自动化执行）
- **范围**：所有RESTful API接口，包括正常请求、边界值、异常参数、权限校验等场景
- **重点**：微服务间调用链路、HL7消息解析接口、AI辅助诊断接口
- **执行方式**：编写Postman测试集合，通过Newman实现自动化回归

#### 2.1.3 功能测试

- **工具**：Selenium WebDriver + ChromeDriver
- **范围**：前端页面交互功能、业务流程端到端测试
- **重点**：用户登录流程、标本全生命周期管理、检验申请到报告生成完整流程
- **执行方式**：编写自动化测试脚本，覆盖主要业务场景

#### 2.1.4 性能测试

- **工具**：Apache JMeter 5.6
- **范围**：系统整体并发性能、各微服务响应时间、数据库查询性能
- **性能指标**：
  - 日常场景：50人并发，平均响应时间不超过3秒
  - 高峰场景：100人并发，平均响应时间不超过5秒
  - 系统可用性不低于99%
- **执行方式**：设计多种负载场景，逐步增加并发用户数，监控系统资源消耗

### 2.2 测试阶段划分

```
单元测试 → 接口测试 → 功能测试 → 性能测试 → 回归测试 → 验收测试
```

各阶段之间存在迭代关系，后一阶段发现的问题可能需要返回前一阶段进行修复和验证。

## 7 缺陷管理

### 7.1 缺陷等级定义

| 等级 | 名称 | 定义 | 处理时限 |
|------|------|------|---------|
| P0 | 致命 | 系统崩溃、数据丢失、核心功能完全不可用 | 4小时内修复 |
| P1 | 严重 | 核心功能异常、数据计算错误、安全漏洞 | 8小时内修复 |
| P2 | 一般 | 非核心功能异常、界面显示错误、性能不达标 | 24小时内修复 |
| P3 | 轻微 | 界面样式问题、文案错误、操作体验不佳 | 48小时内修复 |
| P4 | 建议 | 优化建议、功能改进意见 | 评估后纳入后续版本 |

### 7.2 缺陷处理流程

```
发现缺陷 → 提交缺陷单 → 缺陷评审（确认/拒绝） → 分配修复人 → 缺陷修复 → 回归验证 → 关闭缺陷
```

**详细说明**：

1. **发现缺陷**：测试人员在测试执行过程中发现系统行为与预期不符时，记录缺陷信息
2. **提交缺陷单**：在缺陷管理工具中填写完整的缺陷信息，包括标题、描述、复现步骤、截图/日志、环境信息
3. **缺陷评审**：测试负责人对缺陷进行评审，确认是否为有效缺陷，确定缺陷等级
4. **分配修复人**：根据缺陷所属模块，将缺陷分配给对应的开发人员
5. **缺陷修复**：开发人员分析缺陷原因，进行代码修复并提交
6. **回归验证**：测试人员验证缺陷是否已修复，同时确认未引入新的问题
7. **关闭缺陷**：验证通过后关闭缺陷；若未修复则重新打开，返回开发人员处理

### 7.3 缺陷跟踪工具

本项目使用禅道（ZenTao）开源版作为缺陷管理工具，统一管理缺陷的生命周期。

## 8 风险与应对

| 序号 | 风险描述 | 风险等级 | 影响分析 | 应对措施 |
|------|---------|---------|---------|---------|
| 1 | 微服务间通信异常导致功能不可用 | 高 | 系统整体功能受影响 | 实现服务降级与熔断机制（Sentinel），配置超时重试策略，准备Mock数据 |
| 2 | HL7消息格式不兼容导致对接失败 | 高 | 与外部系统无法正常通信 | 提前与对接方确认消息规范，准备多种格式样本进行兼容性测试 |
| 3 | AI模型准确率不达标 | 中 | 辅助诊断功能可信度降低 | 收集充足训练数据，进行多轮模型调优，设置人工审核环节 |
| 4 | 性能测试环境与生产环境差异大 | 中 | 性能测试结果不能真实反映生产情况 | 尽量模拟生产环境配置，预留性能调优时间 |
| 5 | 测试数据不足影响覆盖度 | 中 | 部分场景无法充分测试 | 编写数据生成脚本，提前准备充足的测试数据 |
| 6 | 测试进度因缺陷修复延迟 | 中 | 整体项目进度受影响 | 合理安排测试与修复并行工作，预留缓冲时间 |
| 7 | 测试人员对业务不熟悉 | 低 | 测试用例设计不完善 | 组织业务培训，邀请领域专家参与用例评审 |
