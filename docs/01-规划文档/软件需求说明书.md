## 1. 引言

### 1.1 编写目的

本软件需求说明书（Software Requirements Specification，简称SRS）是"基于微服务架构的实验室管理系统设计与实现"毕业设计项目的需求规格定义文档。其编写目的在于：

1. **明确系统需求边界**：对系统需要实现的功能需求和非功能需求进行全面、准确、无歧义的描述，为后续的系统设计、编码实现和测试验收提供统一的基准和依据。
2. **指导系统设计与开发**：为架构设计和详细设计提供输入，确保设计人员充分理解业务需求，避免因需求理解偏差导致的返工。
3. **建立验收标准**：为系统测试和项目验收提供可量化的评判标准，确保最终交付的系统满足预期的功能和性能要求。
4. **促进沟通协作**：作为开发者、指导教师和评审人员之间的沟通桥梁，确保各方对系统需求达成一致理解。

本文档的预期读者包括：项目开发者（含AI子Agent团队）、项目指导教师、毕业设计评审专家以及后续可能的系统维护人员。

### 1.2 项目背景

随着医疗信息化建设的深入推进和医疗体制改革的持续深化，医院对实验室信息管理系统（Laboratory Information System，简称LIS）的需求日益增长。LIS系统作为连接临床诊疗与实验室检验的核心枢纽，承担着标本全生命周期管理、检验结果处理、报告审核发布、设备联机通讯等关键业务职能，是现代化医院信息化体系不可或缺的组成部分。

然而，当前许多医疗机构使用的LIS系统仍存在以下突出问题：

- **架构陈旧**：传统LIS系统多采用单体架构，系统各模块之间耦合度高，牵一发而动全身，难以适应业务需求的快速变化和系统的持续迭代升级。
- **功能局限**：部分系统功能覆盖不全面，缺乏危急值自动报警、AI辅助诊断等智能化功能，无法有效辅助临床决策。
- **集成困难**：与医院信息系统（HIS）的对接不够规范，缺乏标准化的接口协议支持，数据共享和业务协同效率低下。
- **用户体验差**：操作界面设计不够友好，学习成本高，影响医护人员的工作效率。

基于上述背景，本项目提出采用微服务架构设计并实现一套功能完善、架构先进、用户体验良好的实验室管理系统。系统参考卫宁健康LIS 6.0产品的功能设计理念，结合Spring Cloud Alibaba微服务技术栈和Vue.js前端框架，构建一套覆盖用户管理、标本管理、检验管理、设备管理、HL7接口对接、数据统计和AI辅助诊断七大核心功能模块的综合性LIS平台，旨在为医疗机构提供高效、安全、智能的实验室信息化解决方案。

## 2. 总体描述

### 2.1 产品视角

本系统是一个基于微服务架构的B/S（浏览器/服务器）模式的实验室信息管理系统，面向医疗机构实验室的日常检验业务流程，提供从标本登记、签收、入库、检验到报告审核发布的全流程数字化管理能力。

**系统定位**：本系统定位为中小型医疗机构实验室的信息化管理平台，参考卫宁健康LIS 6.0产品的功能设计理念，在功能完整性和业务流程规范性方面向行业标杆看齐，同时借助微服务架构和AI辅助诊断等先进技术手段，在系统架构先进性和智能化水平方面实现差异化优势。

**系统组成**：本系统由以下七个核心功能模块构成：

1. **用户管理模块**：提供系统用户的全生命周期管理，包括用户注册登录、角色权限分配、部门组织管理等基础功能，是系统安全运行的基础保障。
2. **标本管理模块**：实现标本从登记到完成的全流程管理，支持条码化管理、扫码签收、状态流转追溯等功能，是实验室日常业务的核心支撑。
3. **检验管理模块**：覆盖检验申请、结果录入、报告审核发布、危急值管理等检验业务全链条，是系统业务价值的核心体现。
4. **设备管理模块**：提供检验设备的台账管理、联机通讯、校准维护等功能，保障检验设备的正常运行和数据准确采集。
5. **HL7接口对接模块**：实现与HIS系统的标准化数据对接，支持患者信息同步、医嘱下载、结果回传等跨系统协同功能。
6. **数据统计模块**：提供多维度的业务数据统计分析和可视化展示，为实验室管理决策提供数据支撑。
7. **AI辅助诊断模块**：基于检验结果数据，提供血常规、尿常规、肝功能等检验项目的智能分析，辅助检验医师进行结果判读和异常识别。

**微服务架构划分**：系统后端采用Spring Cloud Alibaba微服务架构，按照业务领域划分为以下独立微服务：

| 微服务名称 | 职责说明 | 端口 |
|-----------|---------|------|
| lis-gateway | API网关，负责路由转发、统一鉴权、限流熔断 | 8080 |
| lis-auth | 认证授权服务，负责用户登录认证和JWT令牌管理 | 8081 |
| lis-user | 用户管理服务，负责用户、角色、权限、部门管理 | 8082 |
| lis-specimen | 标本管理服务，负责标本全生命周期管理 | 8083 |
| lis-report | 检验管理服务，负责检验申请、结果录入、报告管理 | 8084 |
| lis-equipment | 设备管理服务，负责设备信息维护和联机通讯 | 8085 |
| lis-hl7 | HL7接口服务，负责与HIS系统的消息对接 | 8086 |
| lis-statistics | 数据统计服务，负责业务数据统计与可视化 | 8087 |
| lis-ai | AI辅助诊断服务，负责检验结果的智能分析 | 8088 |

### 2.4 设计约束

1. **技术栈约束**：系统后端必须采用Spring Cloud Alibaba微服务技术栈，前端必须采用Vue.js + Element UI框架，数据库采用MySQL + Redis组合，部署方式采用Docker容器化部署。
2. **架构约束**：系统必须采用微服务架构，各业务模块拆分为独立的微服务，通过Nacos进行服务注册与发现，通过Sentinel实现流量控制与熔断降级。
3. **标准约束**：与HIS系统的对接必须遵循HL7 v2.x消息标准，确保消息格式的规范性和互操作性。
4. **安全约束**：系统必须实现JWT认证机制，敏感数据必须进行AES加密存储，必须防范SQL注入、XSS跨站脚本和CSRF跨站请求伪造等常见安全威胁。
5. **开发周期约束**：作为毕业设计项目，开发周期有限，需要在保证核心功能完整性的前提下合理控制开发范围。
6. **部署约束**：系统必须支持Docker容器化部署，支持一键部署和快速环境搭建。

## 3. 功能需求

### 3.1 用户管理模块

用户管理模块是系统的基础支撑模块，负责系统用户的身份认证、信息管理和权限控制，确保系统安全有序运行。

#### 3.1.1 用户注册

| 项目 | 描述 |
|------|------|
| 功能描述 | 提供新用户账号的注册功能，支持管理员批量创建和用户自助注册两种方式 |
| 输入 | 用户名（4-20位字母数字组合）、密码（8-20位，含大小写字母和数字）、确认密码、真实姓名、工号、所属部门、联系电话、邮箱 |
| 输出 | 注册成功提示信息、用户ID、注册时间；注册失败时返回具体错误原因 |
| 业务规则 | 1. 用户名必须唯一，不可与已注册用户名重复；2. 密码强度必须满足复杂度要求（至少包含大写字母、小写字母和数字）；3. 工号必须唯一，与所在部门关联；4. 新注册用户默认状态为"待审核"，需管理员审核通过后方可使用；5. 用户名和密码不可包含特殊字符和空格 |

#### 3.1.2 用户登录

| 项目 | 描述 |
|------|------|
| 功能描述 | 提供用户身份认证功能，验证用户身份后颁发JWT令牌 |
| 输入 | 用户名、密码、图形验证码（连续输入错误3次后启用） |
| 输出 | 登录成功：JWT访问令牌（access_token，有效期2小时）、刷新令牌（refresh_token，有效期7天）、用户基本信息和角色权限列表；登录失败：错误提示信息 |
| 业务规则 | 1. 连续输入密码错误5次，账号自动锁定30分钟；2. JWT令牌过期后，系统自动使用刷新令牌获取新的访问令牌，无需重新登录；3. 同一账号不允许多处同时登录，后登录者自动挤退先登录者；4. 管理员可手动解锁被锁定的账号；5. 登录日志需记录登录时间、IP地址、登录结果 |

#### 3.1.3 用户信息管理

| 项目 | 描述 |
|------|------|

### 3.2 标本管理模块

标本管理模块是实验室业务流程的核心模块，负责管理标本从登记到检验完成的全生命周期，确保标本信息的完整性和可追溯性。

#### 3.2.1 标本登记

| 项目 | 描述 |
|------|------|
| 功能描述 | 记录标本的基本信息，建立标本与患者、检验项目的关联关系 |
| 输入 | 患者信息（患者ID/姓名/住院号/门诊号）、标本类型（血液/尿液/粪便/体液等）、检验项目（可多选）、采集时间、采集人、送检科室、临床诊断、备注 |
| 输出 | 标本编号、条码信息、登记成功提示、标本标签打印任务 |
| 业务规则 | 1. 标本编号由系统自动生成，格式为"标本类型代码+年月日+6位流水号"；2. 同一患者可同时登记多个标本，每个标本可关联多个检验项目；3. 标本类型与检验项目存在对应关系，系统需校验匹配性；4. 患者信息优先从HIS系统同步获取，如HIS不可用则支持手工录入；5. 登记成功后自动生成条码，状态置为"已登记" |

#### 3.2.2 标本签收

| 项目 | 描述 |
|------|------|
| 功能描述 | 实验室接收标本时进行签收确认，支持扫码签收和批量签收两种模式 |
| 输入 | 扫码签收：条码扫描输入；批量签收：勾选多个待签收标本，统一确认签收 |
| 输出 | 签收成功提示、签收时间记录、标本状态更新为"已签收"、签收人记录 |
| 业务规则 | 1. 扫码签收时，系统自动识别条码并显示标本信息，操作员确认后完成签收；2. 批量签收时，支持按送检科室、送检时间等条件筛选待签收标本；3. 签收时系统自动检查标本是否在有效时间范围内（从采集到签收不超过规定时长），超时标本需注明原因；4. 签收操作需记录签收人、签收时间；5. 签收失败的标本需记录失败原因 |

#### 3.2.3 标本入库

| 项目 | 描述 |
|------|------|
| 功能描述 | 将已签收的标本进行排号入库，分配到具体检验岗位或设备 |
| 输入 | 待入库标本列表、检验分组/岗位分配、优先级设置 |
| 输出 | 入库排号结果、标本状态更新为"已入库"、入库时间记录、排号信息 |
| 业务规则 | 1. 标本入库前必须已完成签收；2. 系统根据检验项目自动推荐检验分组，操作员可手动调整；3. 支持设置优先级（普通/加急/紧急），紧急标本优先排号；4. 支持撤销入库操作，撤销后标本状态回退至"已签收"；5. 撤销入库需记录撤销原因和操作人 |

#### 3.2.4 标本状态流转

| 项目 | 描述 |
|------|------|
| 功能描述 | 管理标本在检验流程中的状态变更，确保状态流转的有序性和可追溯性 |
| 输入 | 状态变更触发条件（如开始检验、完成检验等操作） |
| 输出 | 标本当前状态、状态变更记录（含变更时间、操作人、变更原因） |
| 业务规则 | 1. 标本状态严格按照以下顺序流转：已登记→已签收→已入库→检验中→已完成；2. 每次状态变更需记录时间戳、操作人和变更原因；3. 状态仅允许向前流转，特殊情况下支持管理员回退操作，回退操作需记录原因并通知相关人员；4. 标本状态变更实时推送到相关人员的消息中心 |

#### 3.2.5 标本全流程追溯

| 项目 | 描述 |
|------|------|
| 功能描述 | 提供标本从登记到完成的完整时间线追溯，支持按标本编号或条码查询 |
| 输入 | 标本编号或条码 |
| 输出 | 标本基本信息、完整的状态变更时间线（含各节点时间、操作人、操作内容）、关联的检验报告信息 |
| 业务规则 | 1. 追溯信息按时间倒序排列，最新操作显示在最前；2. 支持查看每个时间节点的详细操作信息；3. 追溯数据不可篡改，仅支持追加；4. 支持导出追溯记录为PDF格式 |

#### 3.2.6 条码管理

| 项目 | 描述 |
|------|------|
| 功能描述 | 管理标本条码的生成、打印和补打 |
| 输入 | 标本编号（补打时需要）、打印份数、打印模板选择 |
| 输出 | 条码标签（含标本编号、患者信息、标本类型、检验项目、条形码/二维码） |
| 业务规则 | 1. 条码在标本登记时自动生成，采用Code 128或QR Code编码格式；2. 条码内容包含标本唯一标识，确保扫码识别的准确性；3. 支持条码补打功能，补打操作需记录补打次数和补打人；4. 条码标签模板支持自定义配置，可调整标签尺寸、内容布局和打印样式；5. 支持批量打印多个标本的条码标签 |

#### 3.2.7 不合格标本管理

| 项目 | 描述 |
|------|------|
| 功能描述 | 记录和处理不合格标本，支持不合格标本的登记、原因标注和重新采集通知 |
| 输入 | 标本编号、不合格原因（溶血/脂血/量不足/容器错误/信息不符/其他）、备注说明、处理方式（退回重新采集/标注后继续检验） |
| 输出 | 不合格标本记录、退回通知（发送至送检科室）、不合格率统计 |
| 业务规则 | 1. 不合格标本需在签收环节或检验前识别并登记；2. 不合格原因必须选择系统预设选项，"其他"选项需填写具体说明；3. 选择"退回重新采集"时，系统自动向送检科室发送退回通知；4. 不合格标本信息纳入统计模块，用于计算不合格率；5. 同一患者连续出现不合格标本时，系统自动预警 |

### 3.3 检验管理模块

检验管理模块是系统的核心业务模块，覆盖检验申请、结果录入、报告审核发布、危急值管理等检验业务全链条。

#### 3.3.1 检验申请管理

| 项目 | 描述 |
|------|------|
| 功能描述 | 管理检验申请单的创建和查询，支持从HIS系统自动获取医嘱和手工创建两种方式 |
| 输入 | 自动获取：HIS系统推送的ORM消息；手工创建：患者信息、检验项目、申请科室、申请医生、临床诊断、申请时间、标本类型、备注 |
| 输出 | 申请单编号、申请单列表（支持多条件查询）、申请单详情 |
| 业务规则 | 1. 优先通过HL7接口从HIS系统获取检验医嘱，HIS不可用时支持手工创建；2. 申请单编号由系统自动生成，格式为"AP+年月日+6位流水号"；3. 同一患者可同时存在多个有效申请单；4. 申请单状态包括：待处理、已关联标本、检验中、已完成、已取消；5. 超过24小时未关联标本的申请单自动标记为"超时预警" |

#### 3.3.2 检验结果录入

| 项目 | 描述 |
|------|------|
| 功能描述 | 提供检验结果的录入功能，支持手工录入和仪器自动导入两种方式 |
| 输入 | 手工录入：标本编号、检验项目、检验结果值、结果单位、参考范围、异常标记、备注；仪器导入：设备通讯接口自动传输的检验结果数据 |
| 输出 | 检验结果记录、异常指标自动标记、结果保存确认 |
| 业务规则 | 1. 手工录入时，系统根据患者性别、年龄自动匹配参考范围；2. 结果超出参考范围时，系统自动标记为异常（高/低），并以醒目颜色显示；3. 仪器导入的结果自动关联标本编号和检验项目，导入后需操作员确认；4. 同一检验项目仅允许录入一次结果，修改已录入结果需记录修改原因；5. 结果录入完成后，标本状态自动更新为"已完成" |

#### 3.3.3 报告审核

| 项目 | 描述 |
|------|------|
| 功能描述 | 提供检验报告的多级审核机制，确保报告的准确性和规范性 |
| 输入 | 待审核报告列表、审核意见、审核结果（通过/退回修改） |
| 输出 | 审核通过：报告状态更新、审核时间记录；审核退回：退回原因说明、报告状态回退 |
| 业务规则 | 1. 报告审核采用两级审核机制：初审（检验医师）和终审（审核医师/主管）；2. 初审通过后报告进入终审队列，终审通过后报告方可发布；3. 审核退回时必须填写退回原因，报告状态回退至"待录入"或"待初审"；4. 审核人和审核时间自动记录，不可篡改；5. 同一审核人不可对同一报告进行初审和终审（需不同人员操作） |

#### 3.3.4 报告发布与回收

| 项目 | 描述 |
|------|------|
| 功能描述 | 管理检验报告的发布和回收操作 |
| 输入 | 报告编号、发布/回收操作、回收原因 |
| 输出 | 报告状态更新、发布时间/回收时间记录、临床通知（发布时） |
| 业务规则 | 1. 报告必须经过终审通过后方可发布；2. 报告发布后，临床医生端可查看报告内容；3. 已发布的报告支持回收操作，回收后临床端不可查看；4. 报告回收需填写回收原因，回收后的报告需重新走审核流程；5. 报告发布后通过HL7接口自动将结果回传至HIS系统 |

#### 3.3.5 报告打印

| 项目 | 描述 |
|------|------|
| 功能描述 | 提供检验报告的打印功能，支持单个打印和批量打印 |
| 输入 | 报告编号（单个打印）、筛选条件（批量打印：时间范围、科室、状态等）、打印模板选择 |
| 输出 | 格式化的检验报告（PDF或直接打印） |
| 业务规则 | 1. 仅已发布状态的报告支持打印；2. 报告打印模板支持自定义配置，可调整报告格式、Logo、页眉页脚等；3. 批量打印时系统自动合并多个报告，减少打印次数；4. 每次打印操作需记录打印时间、打印人和打印份数；5. 支持报告补打功能，补打报告自动加盖"补打"标识 |

#### 3.3.6 报告查询

| 项目 | 描述 |
|------|------|

| 输出 | 报告列表（分页展示）、报告详情（含患者信息、标本信息、检验结果、参考范围、审核信息） |

#### 3.3.7 危急值管理

| 项目 | 描述 |

### 3.5 HL7接口对接模块

HL7接口对接模块负责实现本系统与HIS系统之间的标准化数据交换，是系统集成的关键模块。

#### 3.5.1 HL7 v2.x消息解析

| 项目 | 描述 |
|------|------|
| 功能描述 | 解析和处理HL7 v2.x标准格式的消息 |
| 输入 | HL7 v2.x格式消息字符串（ORM、ORU、ACK等消息类型） |
| 输出 | 解析后的结构化数据、ACK确认消息 |
| 业务规则 | 1. 支持HL7 v2.3、v2.4、v2.5.1等多个版本的消息解析；2. 消息解析采用管道符分隔格式，按Segment（段）、Field（字段）、Component（组件）、Subcomponent（子组件）四级结构解析；3. 解析失败时返回错误类型的ACK消息（AE/AR）；4. 解析成功后返回确认类型的ACK消息（AA）；5. 消息解析过程记录详细日志 |

#### 3.5.2 HIS系统对接

| 项目 | 描述 |
|------|------|
| 功能描述 | 实现与HIS系统的业务数据对接 |
| 输入 | HIS系统发送的ORM消息（检验医嘱）、本系统发送的ORU消息（检验结果回传） |
| 输出 | 患者信息同步结果、医嘱下载结果、检验结果回传确认 |
| 业务规则 | 1. **患者信息同步**：接收HIS系统推送的患者基本信息（ORM消息中的PID段），自动创建或更新本地患者记录；2. **医嘱下载**：接收HIS系统推送的检验医嘱（ORM消息中的ORC/OBR段），自动创建检验申请单；3. **结果回传**：检验报告发布后，系统自动将检验结果组装为ORU消息回传至HIS系统；4. 所有对接操作需记录接口调用日志，包含消息内容、处理结果和处理时间；5. HIS系统不可用时，消息进入等待队列，恢复后自动重发 |

#### 3.5.3 消息异常处理与重试

| 项目 | 描述 |
|------|------|
| 功能描述 | 处理HL7消息传输过程中的异常情况，支持自动重试机制 |
| 输入 | 异常消息信息、重试策略配置（最大重试次数、重试间隔） |
| 输出 | 异常处理记录、重试结果、人工干预提醒 |
| 业务规则 | 1. 消息发送失败时，系统自动按照配置的重试策略进行重试（默认最大重试3次，间隔分别为1分钟、5分钟、30分钟）；2. 超过最大重试次数仍未成功的消息，转入异常队列等待人工处理；3. 消息重复时，系统根据消息控制ID（MSH-10）进行去重处理；4. 异常消息支持人工重新发送和作废操作；5. 系统提供消息监控面板，实时展示消息处理状态统计 |

#### 3.5.4 接口日志记录

| 项目 | 描述 |
|------|------|
| 功能描述 | 记录所有HL7接口的调用日志，便于问题排查和审计 |
| 输入 | 接口调用信息（消息内容、调用方向、处理结果、耗时） |
| 输出 | 接口日志列表、日志详情、日志统计报表 |
| 业务规则 | 1. 日志记录包含：消息唯一标识、消息类型、消息方向（接收/发送）、原始消息内容、处理结果、处理时间、耗时；2. 日志保留期限不少于180天；3. 支持按时间范围、消息类型、处理结果等条件检索日志；4. 敏感信息（如患者身份证号）在日志中脱敏显示 |

### 3.7 AI辅助诊断模块

AI辅助诊断模块基于检验结果数据，提供智能化的结果分析和辅助诊断建议，帮助检验医师提高工作效率和诊断准确性。

#### 3.7.1 血常规结果分析

| 项目 | 描述 |
|------|------|
| 功能描述 | 对血常规检验结果进行智能分析，识别异常指标并提供解读 |
| 输入 | 血常规检验结果数据（白细胞WBC、红细胞RBC、血红蛋白HGB、血小板PLT、中性粒细胞、淋巴细胞、单核细胞、嗜酸性粒细胞、嗜碱性粒细胞、红细胞压积HCT、平均红细胞体积MCV、平均红细胞血红蛋白量MCH、平均红细胞血红蛋白浓度MCHC等） |
| 输出 | 各指标分析结果（正常/偏高/偏低）、异常指标高亮标注、指标组合分析结论、可能的临床意义解释 |
| 业务规则 | 1. 系统根据患者性别、年龄自动匹配参考范围；2. 单项指标异常时，给出该指标异常的常见临床意义；3. 多项指标联合异常时，给出综合分析结论（如"白细胞升高伴中性粒细胞比例增高，提示细菌感染可能"）；4. 分析结果标注置信度等级（高/中/低）；5. 分析结果仅供参考，最终诊断由检验医师确认 |

#### 5.4.1 微服务间通信

- **同步通信**：微服务间的同步调用采用OpenFeign声明式HTTP客户端，配合Nacos服务发现实现负载均衡
- **异步通信**：非实时性业务（如消息通知、日志记录）采用消息队列异步处理
- **服务熔断**：通过Sentinel实现服务调用的熔断降级，防止级联故障

#### 5.4.2 前后端通信

- **通信协议**：前后端采用HTTP/HTTPS协议通信，数据格式为JSON
- **API规范**：后端API遵循RESTful设计规范，使用标准HTTP方法（GET/POST/PUT/DELETE）
- **统一响应格式**：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {},
  "timestamp": 1712345678000
}
```

- **认证方式**：请求头携带JWT令牌，格式为`Authorization: Bearer <token>`
- **跨域处理**：后端配置CORS跨域策略，允许前端域名访问

6. **标本-检验报告**：标本与检验报告为一对一关系，一个标本生成一份检验报告
7. **设备-检验结果**：设备与检验结果为一对多关系，一台设备可产生多条检验结果

## 7. 附录

### 7.1 数据流图描述

#### 7.1.1 顶层数据流图

系统顶层数据流描述了LIS系统与外部实体之间的数据交互关系：

**外部实体**：
- **临床医生**：输入检验申请，接收检验报告
- **护士**：输入标本信息，接收条码标签
- **检验医师**：输入检验结果，执行报告审核
- **HIS系统**：交换患者信息、检验医嘱、检验结果
- **检验设备**：传输检验数据

**主要数据流**：
1. 临床医生 → LIS系统：检验申请信息
2. LIS系统 → 护士：条码标签
3. 护士 → LIS系统：标本登记信息
4. 检验设备 → LIS系统：检验结果数据
5. 检验医师 → LIS系统：审核意见、手工录入结果
6. LIS系统 → 临床医生：检验报告
7. HIS系统 → LIS系统：患者信息、检验医嘱（HL7 ORM消息）
8. LIS系统 → HIS系统：检验结果（HL7 ORU消息）

#### 7.1.2 标本管理数据流图（0层）

**处理过程**：
1. **标本登记**：接收患者信息和检验项目，生成标本编号和条码，建立标本记录
2. **标本签收**：接收扫码/批量签收请求，校验标本有效性，更新标本状态
3. **标本入库**：接收入库请求，分配检验分组和排号，更新标本状态
4. **条码管理**：生成条码，管理条码打印和补打

**数据流**：
- 患者信息 + 检验项目 → 标本登记 → 标本记录 + 条码信息
- 条码信息 → 标本签收 → 签收记录
- 签收记录 → 标本入库 → 入库记录 + 排号信息
- 标本记录 → 条码管理 → 条码标签

**数据存储**：
- D1：标本信息表（存储标本基本信息和状态）
- D2：条码记录表（存储条码生成和打印记录）
- D3：患者信息表（存储患者基本信息）

#### 7.1.3 检验管理数据流图（0层）

**处理过程**：
1. **检验申请处理**：接收检验医嘱，创建检验申请单
2. **结果录入**：接收仪器数据或手工录入，保存检验结果
3. **报告审核**：执行初审和终审，验证结果准确性
4. **报告发布**：发布审核通过的报告，回传结果至HIS
5. **危急值处理**：识别危急值，发送报警通知，记录处理过程

**数据流**：
- 检验医嘱 → 检验申请处理 → 检验申请单
- 仪器数据/手工录入 → 结果录入 → 检验结果
- 检验结果 → 报告审核 → 审核意见
- 审核通过的报告 → 报告发布 → 检验报告 + HL7 ORU消息
- 检验结果 → 危急值处理 → 报警通知 + 处理记录

**数据存储**：
- D4：检验申请表
- D5：检验结果表
- D6：检验报告表
- D7：危急值记录表
