## 一、系统概述

### 1.1 系统简介

实验室信息管理系统（LIS）是基于微服务架构设计开发的综合性实验室管理平台。系统采用前后端分离架构，后端基于 Spring Cloud Alibaba 构建 9 个独立微服务，前端基于 Vue.js 3 构建单页应用。系统通过 Docker 容器化部署，支持一键启动和弹性扩展。

### 1.2 系统架构描述

系统整体架构分为以下层次：

**1. 接入层**
- Nginx 反向代理服务器：负责 HTTPS 终结、静态资源服务、负载均衡
- 端口：80（HTTP）、443（HTTPS）

**2. 网关层**
- Gateway 网关服务：负责 API 路由转发、统一鉴权、限流熔断
- 端口：8080

**3. 服务层**
- 9 个业务微服务，各服务独立部署、独立数据库
- 端口范围：9100-9800

**4. 数据层**
- MySQL 8.0：业务数据持久化存储
- Redis 6.x：缓存、会话管理、分布式锁
- RocketMQ：异步消息队列

**5. 基础设施层**
- Nacos：服务注册与配置中心
- Sentinel：流量控制与熔断降级
- Seata：分布式事务管理
- ELK（Elasticsearch + Logstash + Kibana）：日志收集与分析

**6. 部署层**
- Docker + Docker Compose：容器化部署与编排

|---------|------|--------|---------|
| lis-gateway | 8080 | 无 | API 网关，路由转发与鉴权 |
| lis-auth | 9100 | lis_auth | 用户认证与令牌管理 |

## 三、系统启动与停止

### 3.1 启动顺序

系统各组件之间存在依赖关系，必须按照以下顺序启动：

```
第1步：基础设施组件
  ├── MySQL 数据库
  ├── Redis 缓存
  └── RocketMQ 消息队列

第2步：注册与配置中心
  └── Nacos Server

第3步：微服务（按顺序启动）
  ├── lis-auth（认证服务）
  ├── lis-user（用户服务）
  ├── lis-specimen（标本服务）
  ├── lis-report（报告服务）
  ├── lis-equipment（设备服务）
  ├── lis-hl7（HL7 服务）
  ├── lis-statistics（统计服务）
  ├── lis-ai（AI 服务）
  └── lis-gateway（网关服务，最后启动）

第4步：前端与代理
  └── Nginx（反向代理 + 前端静态资源）
```

**Docker Compose 一键启动：**

```bash
# 进入项目部署目录
cd /opt/lis-deploy

# 启动所有服务
docker-compose up -d

# 查看服务状态
docker-compose ps

# 查看服务日志
docker-compose logs -f
```

**单独启动某个服务：**

```bash
# 启动指定服务
docker-compose up -d lis-auth

# 重启指定服务
docker-compose restart lis-auth
```

### 3.2 停止顺序

系统停止时应按照启动的逆序进行：

```
第1步：停止前端与代理
  └── Nginx

第2步：停止网关
  └── lis-gateway

第3步：停止业务微服务（顺序不限）
  ├── lis-auth
  ├── lis-user
  ├── lis-specimen
  ├── lis-report
  ├── lis-equipment
  ├── lis-hl7
  ├── lis-statistics
  └── lis-ai

第4步：停止注册与配置中心
  └── Nacos Server

第5步：停止基础设施组件
  ├── RocketMQ
  ├── Redis
  └── MySQL
```

**Docker Compose 一键停止：**

```bash
# 停止所有服务
docker-compose down

# 停止所有服务并删除数据卷（慎用）
docker-compose down -v
```

## 四、日常维护

### 4.1 数据库备份

**自动备份脚本（建议配置 crontab 定时执行）：**

```bash
#!/bin/bash
# 数据库备份脚本
# 保存路径：/opt/lis-deploy/backup/mysql/

BACKUP_DIR="/opt/lis-deploy/backup/mysql"
DATE=$(date +%Y%m%d_%H%M%S)
MYSQL_CONTAINER="lis-mysql"
MYSQL_USER="root"
MYSQL_PASSWORD="your_password"

# 备份所有 LIS 数据库
DATABASES="lis_auth lis_user lis_specimen lis_report lis_equipment lis_hl7 lis_statistics lis_ai"

for DB in $DATABASES; do
    docker exec $MYSQL_CONTAINER mysqldump -u$MYSQL_USER -p$MYSQL_PASSWORD \
        --single-transaction --routines --triggers $DB \
        | gzip > $BACKUP_DIR/${DB}_${DATE}.sql.gz
done

# 删除 30 天前的备份文件
find $BACKUP_DIR -name "*.sql.gz" -mtime +30 -delete

echo "Backup completed at $DATE"
```

**定时任务配置（每天凌晨 2 点执行）：**

```bash
crontab -e
# 添加以下行
0 2 * * * /opt/lis-deploy/scripts/mysql_backup.sh >> /opt/lis-deploy/logs/backup.log 2>&1
```

**手动恢复数据库：**

```bash
# 解压备份文件
gunzip lis_user_20250301_020000.sql.gz

# 恢复数据库
docker exec -i lis-mysql mysql -uroot -pyour_password lis_user < lis_user_20250301_020000.sql
```

## 五、故障处理

### 5.1 常见故障及解决方案

**故障1：服务启动失败**

| 项目 | 内容 |
|------|------|
| 现象 | Docker 容器启动后立即退出，状态为 Exited |
| 排查步骤 | 1. 查看容器日志：`docker-compose logs <服务名>`<br>2. 检查配置文件是否正确<br>3. 检查端口是否被占用<br>4. 检查依赖服务（MySQL、Redis、Nacos）是否正常 |
| 常见原因 | 配置错误、端口冲突、依赖服务未启动、内存不足 |
| 解决方案 | 根据日志信息修正配置，确保依赖服务正常运行 |

**故障2：Nacos 服务注册失败**

| 项目 | 内容 |
|------|------|
| 现象 | 微服务启动后未出现在 Nacos 服务列表中 |
| 排查步骤 | 1. 检查 Nacos 是否正常运行<br>2. 检查微服务的 Nacos 配置（地址、命名空间、分组）<br>3. 检查网络连通性 |
| 常见原因 | Nacos 地址配置错误、网络不通、Nacos 服务端异常 |
| 解决方案 | 检查 bootstrap.yml 中的 Nacos 配置，确保网络可达 |

**故障3：数据库连接失败**

| 项目 | 内容 |
|------|------|
| 现象 | 服务日志中出现 "Communications link failure" 或 "Connection refused" |
| 排查步骤 | 1. 检查 MySQL 容器是否运行<br>2. 检查数据库连接配置（地址、端口、用户名、密码）<br>3. 检查 MySQL 最大连接数是否已满 |
| 常见原因 | MySQL 未启动、连接配置错误、连接池耗尽 |
| 解决方案 | 确保 MySQL 正常运行，检查并调整连接池配置 |

**故障4：Redis 连接异常**

| 项目 | 内容 |
|------|------|
| 现象 | 缓存相关操作报错，或页面加载缓慢 |
| 排查步骤 | 1. 检查 Redis 容器是否运行<br>2. 检查 Redis 内存是否已满<br>3. 检查 Redis 连接配置 |
| 常见原因 | Redis 内存不足、连接配置错误、Redis 服务宕机 |
| 解决方案 | 检查 Redis 内存使用情况，调整 maxmemory 配置 |

**故障5：API 请求返回 502/504**

| 项目 | 内容 |
|------|------|
| 现象 | 前端请求返回 502 Bad Gateway 或 504 Gateway Timeout |
| 排查步骤 | 1. 检查 Gateway 网关是否正常运行<br>2. 检查目标微服务是否注册到 Nacos<br>3. 检查目标微服务是否正常运行<br>4. 检查请求是否触发了 Sentinel 限流 |
| 常见原因 | 目标服务未启动、服务调用超时、触发限流熔断 |
| 解决方案 | 检查目标服务状态，调整超时配置和限流阈值 |

**故障6：文件上传失败**

| 项目 | 内容 |
|------|------|
| 现象 | 上传文件时返回错误，文件无法保存 |
| 排查步骤 | 1. 检查文件存储目录是否存在且有写入权限<br>2. 检查磁盘空间是否充足<br>3. 检查 Nginx 上传文件大小限制 |
| 常见原因 | 存储目录权限不足、磁盘空间不足、文件大小超限 |
| 解决方案 | 修改目录权限（chmod 755），调整 Nginx client_max_body_size 配置 |

**故障7：Docker 容器内存溢出**

| 项目 | 内容 |
|------|------|
| 现象 | 容器频繁重启，日志中出现 OOMKilled |
| 排查步骤 | 1. 查看容器资源使用：`docker stats`<br>2. 查看容器日志中的内存溢出信息<br>3. 检查 JVM 堆内存配置 |
| 常见原因 | JVM 堆内存配置过大、应用内存泄漏、并发量过高 |
| 解决方案 | 调整 Docker 内存限制和 JVM 堆内存参数，排查内存泄漏 |

### 5.2 应急处理流程

```
1. 发现故障
   └── 通过监控告警或用户反馈发现异常

2. 初步判断
   └── 查看日志，确定故障类型和影响范围

3. 应急响应
   ├── 轻微故障：记录问题，安排修复
   ├── 严重故障：立即处理，必要时重启服务
   └── 致命故障：启动应急预案，通知相关人员

4. 故障修复
   └── 根据故障类型执行对应的解决方案

5. 验证恢复
   └── 确认服务恢复正常，业务功能可用

6. 总结记录
   └── 记录故障原因、处理过程和改进措施
```

## 六、升级维护

### 6.1 版本升级流程

**升级前准备：**

1. 阅读新版本的 Release Notes，了解变更内容
2. 在测试环境完成升级测试，验证功能正常
3. 完成数据库和配置文件的备份
4. 通知相关用户即将进行升级维护

**升级步骤：**

```bash
# 1. 备份数据库
/opt/lis-deploy/scripts/mysql_backup.sh

# 2. 拉取最新代码和镜像
cd /opt/lis-deploy
git pull origin main
docker-compose pull

# 3. 停止当前服务
docker-compose down

# 4. 执行数据库升级脚本（如有）
docker exec -i lis-mysql mysql -uroot -p < upgrade/V1.0.0_to_V1.1.0.sql

# 5. 更新 Nacos 配置（如有变更）
# 通过 Nacos 控制台手动更新或使用配置导入功能

# 6. 启动新版本服务
docker-compose up -d

# 7. 验证服务状态
docker-compose ps
curl http://localhost:8080/actuator/health
```

**升级后验证：**

1. 检查所有服务是否正常启动
2. 验证核心业务流程是否正常
3. 检查日志是否有异常信息
4. 进行简单的性能测试

**回滚方案：**

如果升级后出现严重问题，按以下步骤回滚：

```bash
# 1. 停止当前服务
docker-compose down

# 2. 恢复数据库备份
gunzip < /opt/lis-deploy/backup/mysql/lis_user_<备份时间>.sql.gz | \
    docker exec -i lis-mysql mysql -uroot -p lis_user

# 3. 切换到旧版本代码
git checkout <旧版本tag>

# 4. 启动旧版本服务
docker-compose up -d
```

### 6.2 数据库升级

数据库升级脚本统一存放在 `upgrade/` 目录下，按版本号命名：

```
upgrade/
├── V1.0.0_to_V1.1.0.sql
├── V1.1.0_to_V1.2.0.sql
└── README.md
```

升级脚本应遵循以下规范：
- 每个升级脚本必须是幂等的（可重复执行不报错）
- 使用 IF NOT EXISTS 等安全语法
- 包含升级前的数据校验
- 包含升级后的数据验证

**命名空间：**
- 开发环境：`dev`

### 7.4 配置变更流程

1. 在 Nacos 控制台中修改配置
2. 配置变更后，Nacos 会自动推送至各微服务
3. 各微服务通过 `@RefreshScope` 注解实现配置热更新
4. 验证配置是否生效（查看服务日志或调用验证接口）
5. 如配置变更涉及数据库结构，需先执行数据库升级脚本

**注意事项：**
- 生产环境配置变更前必须在测试环境验证
- 配置变更应有记录，包含变更人、变更时间、变更内容
- 敏感配置（密码、密钥）变更后应通知相关人员
- 建议对 Nacos 配置进行版本管理，便于回溯
