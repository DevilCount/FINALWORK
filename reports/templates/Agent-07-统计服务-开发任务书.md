## 二、任务背景

### 2.1 业务背景

数据统计是LIS系统管理决策的重要支撑模块。通过对检验工作量、标本处理量、设备使用情况、报告审核情况等多维度数据的统计分析，实验室管理者可以全面了解实验室运行状况，优化资源配置，提升工作效率。统计服务Agent作为独立的数据统计微服务，负责从各个业务模块采集数据，进行聚合计算和格式化处理，为前端展示提供标准化的统计数据接口。

### 2.2 技术背景

统计服务采用Spring Boot框架构建，通过定时任务自动采集和计算统计数据，使用MyBatis进行多表关联聚合查询，使用Redis缓存统计结果以提升查询性能。图表数据按照ECharts标准格式输出，前端可直接渲染展示。数据导出功能使用EasyExcel库实现高性能Excel文件生成。

### 2.3 核心目标

1. 建立完整的工作量统计体系，支持多维度、多粒度的数据统计
2. 实现标本全流程统计数据采集与分析
3. 提供设备使用情况统计分析
4. 建立报告审核流程的统计监控
5. 输出标准化的ECharts图表数据格式
6. 实现定时自动统计任务，减少人工操作
7. 提供便捷的数据导出功能

## 三、任务清单

### 任务1：工作量统计

#### 详细描述

实现实验室工作量统计功能，支持按时间维度（日/周/月/季度/年）、科室维度、操作员维度进行统计。统计指标包括：标本签收数量、检验完成数量、报告审核数量、报告发布数量、TAT平均时长等。

#### 输入

- **统计请求**：统计类型（按时间/科室/操作员）、时间范围（开始日期、结束日期）、统计粒度（日/周/月/季度/年）、科室ID（可选）、操作员ID（可选）

#### 输出

- **统计结果**：统计数据列表，每条记录包含统计维度值、标本签收量、检验完成量、报告审核量、报告发布量、TAT平均时长等
- **汇总数据**：总计、平均值、最大值、最小值、环比增长率、同比增长率

#### 技术要点

- 使用MyBatis动态SQL实现多维度灵活查询
- 按时间维度统计使用DATE_FORMAT函数进行时间分组
- 按科室维度统计通过JOIN科室表获取科室信息
- 按操作员维度统计通过JOIN用户表获取操作员信息
- TAT（Turnaround Time）计算：从标本签收到报告发布的平均时长，单位为分钟
- 支持同比（与去年同期对比）和环比（与上一周期对比）计算
- 统计结果缓存至Redis，缓存时间根据统计粒度设置（日统计缓存1小时、月统计缓存24小时）
- 大数据量统计使用游标分批处理，避免内存溢出

### 任务2：标本统计

#### 详细描述

实现标本处理全流程的统计分析功能，统计指标包括：标本签收量、检验完成量、不合格标本数量及不合格率、按标本类型分类统计、按检验急缓分类统计等。

#### 输入

- **统计请求**：时间范围、统计维度（按类型/按急缓/按科室/按时间段）、标本类型（可选）、科室ID（可选）

#### 输出

- **标本签收统计**：签收总量、按时间段分布
- **标本完成统计**：完成总量、完成率、按时间段分布
- **不合格标本统计**：不合格数量、不合格率、不合格原因分布（溶血、脂血、量不足、标识不清等）
- **标本类型分布**：各类型标本数量及占比（血液、尿液、粪便、体液等）
- **急缓分类统计**：急诊标本数量、普通标本数量、急诊占比

#### 技术要点

- 不合格率计算：不合格标本数量 / 标本签收总量 * 100%
- 不合格原因分类统计使用GROUP BY聚合
- 标本类型分布使用饼图数据格式输出（{name, value}数组）
- 急缓分类统计支持急诊/普通/加急等多种优先级
- 标本流转效率统计：平均签收时间、平均检验时间、平均报告时间
- 统计数据支持多维度交叉分析（如：科室+标本类型、时间+急缓等）
- 使用数据库索引优化聚合查询性能

### 任务3：设备使用统计

#### 详细描述

实现设备使用情况的统计分析功能，统计指标包括：设备使用次数（检验标本数量）、设备运行时长、设备故障次数、设备故障率、设备使用率等。

#### 输入

- **统计请求**：时间范围、设备ID（可选，不传则统计全部设备）、统计粒度（日/周/月）

#### 输出

- **设备使用次数统计**：各设备在统计周期内的使用次数
- **设备运行时长统计**：各设备的累计运行时长
- **设备故障统计**：故障次数、故障率、故障类型分布
- **设备使用率排名**：按使用率从高到低排序
- **设备效率分析**：单位时间内处理标本数量排名

#### 技术要点

- 设备使用次数通过统计关联到该设备的检验记录数量获得
- 设备运行时长通过设备联机状态日志计算（在线时长累加）
- 设备故障率计算：故障次数 / 使用次数 * 100%
- 设备使用率计算：实际使用时长 / 可用时长 * 100%
- 支持设备间的横向对比分析
- 统计结果使用柱状图数据格式输出（适合多设备对比展示）
- 提供设备使用趋势分析（折线图数据格式）

### 任务4：报告统计

#### 详细描述

实现检验报告的统计分析功能，统计指标包括：报告总数、已审核报告数、已发布报告数、审核率、发布率、审核时长统计、退回修改次数等。

#### 输入

- **统计请求**：时间范围、科室ID（可选）、审核人ID（可选）、统计粒度（日/周/月）

#### 输出

- **报告数量统计**：报告总数、已审核数、已发布数、待审核数
- **报告审核率**：已审核报告数 / 报告总数 * 100%
- **报告发布率**：已发布报告数 / 报告总数 * 100%
- **审核时长统计**：平均审核时长、最长审核时长、最短审核时长
- **退回统计**：退回修改次数、退回率、退回原因分布
- **审核人工作量排名**：按审核报告数量排名

#### 技术要点

- 审核时长计算：从报告生成到审核完成的时长
- 退回率计算：退回修改次数 / 报告总数 * 100%
- 退回原因分类统计（数据异常、信息不全、需要复核等）
- 审核人工作量排名支持按科室筛选
- 报告统计支持按检验项目分类（如：血常规报告数量、生化报告数量等）
- 统计数据支持同比环比分析

### 任务5：ECharts图表数据格式化

#### 详细描述

将统计数据按照ECharts图表库的标准数据格式进行格式化输出，支持折线图、柱状图、饼图、散点图等多种图表类型的数据格式。前端可直接使用格式化后的数据进行图表渲染，无需二次处理。

#### 输入

- **原始统计数据**：聚合查询结果集
- **图表类型**：折线图（line）、柱状图（bar）、饼图（pie）、散点图（scatter）
- **图表配置**：标题、图例、坐标轴标签、数据系列名称等

#### 输出

- **ECharts Option对象**：符合ECharts标准的完整配置对象，包含title、legend、xAxis、yAxis、series等配置项
- **各图表类型数据格式**：
  - 折线图/柱状图：{xAxis: [...], series: [{name, data: [...]}]}
  - 饼图：{series: [{name, data: [{name, value}]}]}
  - 散点图：{series: [{name, data: [[x, y], ...]}]}

#### 技术要点

- 使用策略模式设计不同图表类型的数据格式化器
- 折线图数据格式化器（LineChartFormatter）：处理时间序列数据，X轴为时间，Y轴为数值
- 柱状图数据格式化器（BarChartFormatter）：处理分类对比数据，支持堆叠柱状图
- 饼图数据格式化器（PieChartFormatter）：处理占比数据，计算百分比
- 支持多数据系列（多个Y轴指标同时展示）
- 支持数据聚合（日数据聚合为周数据、周数据聚合为月数据）
- 图表配置支持自定义（颜色、字体、大小等）
- 数据为空时显示友好的提示信息（"暂无数据"）

### 任务6：定时统计任务

#### 详细描述

实现定时自动统计任务，按照预设的时间周期自动执行数据统计并缓存统计结果。支持日统计（每日凌晨执行）、周统计（每周一凌晨执行）、月统计（每月1日凌晨执行）等不同频率的统计任务。

#### 输入

- **任务配置**：统计任务类型（日统计/周统计/月统计）、执行时间（Cron表达式）、统计维度、是否启用

#### 输出

- **统计结果**：计算完成的统计数据，写入统计结果表
- **执行日志**：任务执行时间、执行耗时、执行状态、处理数据量

#### 技术要点

- 使用Spring的@Scheduled注解实现定时任务
- 日统计任务：每日凌晨01:00执行，统计前一天的数据
  - Cron表达式：`0 0 1 * * ?`
- 周统计任务：每周一凌晨02:00执行，统计上周的数据
  - Cron表达式：`0 0 2 ? * MON`
- 月统计任务：每月1日凌晨03:00执行，统计上月的数据
  - Cron表达式：`0 0 3 1 * ?`
- 定时任务使用分布式锁（Redisson）防止多实例重复执行
- 任务执行结果写入work_statistics和specimen_statistics表
- 任务执行异常时记录错误日志并发送告警通知
- 支持手动触发统计任务（管理接口）
- 任务执行超时设置（默认30分钟），超时自动终止
- 统计任务执行记录保留90天

### 任务7：数据导出

#### 详细描述

实现统计数据导出功能，支持将统计数据导出为Excel格式文件。导出内容包括：工作量统计报表、标本统计报表、设备使用统计报表、报告统计报表等。

#### 输入

- **导出请求**：报表类型（工作量/标本/设备/报告）、时间范围、统计维度、导出格式（xlsx）

#### 输出

- **导出文件**：Excel格式文件（.xlsx），包含表头、数据行、合计行
- **导出结果**：文件下载链接或文件流

#### 技术要点

- 使用EasyExcel库实现高性能Excel文件生成
- 支持大数据量导出（使用EasyExcel的流式写入，避免内存溢出）
- Excel模板设计：统一的表头样式（字体、背景色、边框）、数据行样式、合计行样式
- 支持多Sheet页导出（一个报表包含多个Sheet页）
- 导出文件命名规则：报表类型_统计时间范围_导出日期.xlsx
- 导出操作记录日志（导出人、导出时间、导出数据量）
- 导出数据量限制（单次导出最多10万行）
- 异步导出：大数据量导出使用异步任务，完成后通知用户下载
- 导出文件临时存储，保留24小时后自动清理

## 五、技术约束

### 5.1 技术栈要求

| 技术 | 版本要求 | 用途 |
|------|----------|------|
| Java | JDK 17+ | 开发语言 |
| Spring Boot | 2.7.18 | 应用框架 |

| EasyExcel | 3.3.x | Excel导出 |
| Maven | 3.8+ | 构建工具 |

### 5.2 编码规范

- 遵循阿里巴巴Java开发手册编码规范
- 统计服务类命名以"StatisticsService"结尾
- 图表格式化器命名以"ChartFormatter"结尾
- 定时任务类命名以"StatisticsTask"结尾
- 导出处理器命名以"ExportHandler"结尾
- SQL查询需添加索引提示，确保聚合查询性能
- 统计方法需添加性能日志，记录查询耗时

### 5.3 接口规范

- RESTful API设计风格
- 统一响应格式：{code, message, data}
- 接口版本号：/api/statistics
- 图表数据接口统一返回ECharts Option格式
- 导出接口返回文件流（Content-Type: application/vnd.openxmlformats-officedocument.spreadsheetml.sheet）

### 5.4 性能约束

- 统计查询响应时间 <= 3秒（常规统计）
- 统计查询响应时间 <= 10秒（大数据量跨月统计）
- Excel导出响应时间 <= 30秒（10万行数据）
- 定时任务执行时间 <= 30分钟
- 缓存命中率 >= 80%

## 【Git工作流规范】

> 本章节为V3.0新增，所有规则具有强制约束力。详细规范参见 `/workspace/lis-project/docs/05-开发规范/Git版本控制规范.md`。

### 分支管理

- **工作分支**：`feature/agent-07-statistics-service`
- **修复分支**：`bugfix/agent-07-{description}`
- **禁止**直接在`main`或`develop`分支上提交任何代码
- 分支由项目经理（PM）创建，你只能在分配的分支上工作

### 提交规范

- **格式**：`[Agent-07] {类型}({范围}): {简述}`
- **类型**：feat（新功能）、fix（修复）、refactor（重构）、docs（文档）、test（测试）、chore（构建/配置）
- **示例**：`[Agent-07] feat(统计服务): 新增XXX功能`
- 每个功能点至少一次提交，禁止积压大量变更后一次性提交
- 提交前必须确保代码可编译、功能可运行

### 过程文档要求

每个功能点完成后，必须同步提交过程文档：

| 文档类型 | 存放路径 | 提交时机 |
|---------|---------|---------|
| 功能开发记录 | `/workspace/lis-project/docs/process/Agent-07/{功能名称}-开发记录.md` | 每个功能点完成时 |
| 变更日志 | `/workspace/lis-project/docs/process/Agent-07/CHANGELOG.md` | 每次提交时更新 |

过程文档模板参见：`/workspace/lis-project/docs/process/开发记录模板.md`

### 阶段交付要求

完成本阶段全部任务后，必须提交以下交付物（缺一不可）：

1. ✅ 阶段开发报告（按D09《开发报告模板》格式）
2. ✅ CHANGELOG变更日志
3. ✅ 所有功能点的开发记录（`docs/process/Agent-07/`目录下）
4. ✅ 数据库变更脚本（如有数据库结构变更，含回滚脚本）
5. ✅ 通知项目经理进行合并审查

### PM汇报机制

- **每完成一个功能点**：Git commit + 过程文档 + 通知PM（附commit hash）
- **遇到阻塞问题**：立即通知PM，不得自行跳过或绕过
- **每日结束前**：提交当日进度总结到CHANGELOG
- **合并请求**：全部完成后，等待PM审查代码和文档，经PM审批后方可合并到develop

### 违规后果

| 违规行为 | 后果 |
|---------|------|
| 直接在main/develop上提交 | PM执行git reset回退 + 警告 |
| 提交信息不规范 | PM驳回提交，要求修改 |
| 过程文档缺失 | PM驳回合并请求 |
| 未经审批自行合并 | PM回退合并 + 暂停Agent |
| 虚假代码/文档 | PM回退分支 + 按红线禁令R1处理 |
