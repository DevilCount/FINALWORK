## 二、任务背景

### 2.1 业务背景

HL7（Health Level Seven）是医疗信息系统中广泛使用的国际标准通讯协议，用于不同医疗信息系统之间的数据交换。在LIS系统中，HL7接口是实现与HIS（医院信息系统）数据互通的核心通道。通过HL7接口，HIS系统可以向LIS系统发送检验医嘱（ORM消息），LIS系统检验完成后可以向HIS系统回传检验结果（ORU消息），实现检验业务的全流程信息化闭环。

### 2.2 技术背景

HL7 v2.x版本是目前国内医疗机构应用最广泛的HL7标准版本，采用管道符（|）分隔的文本格式。消息传输层使用MLLP（Minimum Lower Layer Protocol）协议，基于TCP/IP实现可靠传输。本服务使用HAPI-HL7开源库进行消息的构建、解析和处理，使用Netty框架实现MLLP协议通讯。

### 2.3 核心目标

1. 实现HL7 v2.x消息的完整解析与构建能力
2. 实现ORM^O01消息处理，接收HIS系统发送的检验医嘱
3. 实现ORU^R01消息处理，向HIS系统回传检验结果
4. 实现ACK消息自动生成与发送，确保消息交互的可靠性
5. 建立MLLP协议通讯层，支持TCP/IP可靠传输
6. 完善消息日志记录与追踪机制
7. 提供灵活的接口配置管理能力
8. 建立完善的异常处理与重试机制

## 三、任务清单

### 任务1：HL7 v2.x消息解析

#### 详细描述

基于HAPI-HL7库实现HL7 v2.x版本消息的解析功能，支持将HL7文本消息解析为结构化的Java对象，同时支持将Java对象构建为HL7文本消息。需要支持HL7 v2.3、v2.4、v2.5.1等常用版本。

#### 输入

- **解析请求**：HL7原始消息文本字符串、HL7版本号
- **构建请求**：消息类型、消息结构数据（Java对象）

#### 输出

- **解析结果**：结构化的HL7消息对象（Message对象），包含MSH、PID、PV1、ORC、OBR、OBX等段信息
- **构建结果**：符合HL7 v2.x标准的文本格式消息字符串
- **解析异常**：解析失败时返回错误码、错误描述、错误位置（段号、字段号）

#### 技术要点

- 使用HAPI-HL7库（ca.uhn.hapi）进行消息解析和构建
- 支持自定义PipeParser配置，处理非标准字符编码
- 实现HL7消息版本自动识别（通过MSH-12字段判断版本）
- 支持消息结构验证（必填段、必填字段校验）
- 实现自定义消息类型扩展（针对HIS系统的私有字段）
- 支持中文编码处理（UTF-8/GBK编码自动检测与转换）
- 解析性能优化：使用对象池复用Parser实例

---

### 任务2：ORM^O01消息处理

#### 详细描述

实现ORM^O01（Order Message）消息的接收与处理功能。ORM^O01消息由HIS系统发送，用于向LIS系统传递检验医嘱信息。处理流程包括：消息接收、消息解析、数据提取、业务校验、数据入库、生成ACK应答。

#### 输入

- **ORM^O01消息**：包含以下段信息
  - MSH：消息头（发送方、接收方、消息类型、消息时间等）
  - PID：患者信息（患者ID、姓名、性别、出生日期等）
  - PV1：就诊信息（就诊号、科室、病房、床号等）
  - ORC：医嘱公共信息（医嘱控制码、医嘱状态等）
  - OBR：检验医嘱信息（检验项目、标本类型、临床医生、优先级等）

#### 输出

- **处理结果**：医嘱接收成功/失败状态
- **入库数据**：检验医嘱记录（写入LIS系统数据库）
- **ACK消息**：自动生成的应答消息（AA-接受、AE-错误、AR-拒绝）

#### 技术要点

- 使用HAPI的ORM_O01模型类进行消息解析
- 实现消息路由：根据ORC-1（Order Control）字段判断操作类型（NW-新建、SC-状态变更、CA-取消、DC-中止）
- 患者信息自动匹配：根据PID-3（患者ID）查询LIS系统患者信息，不存在时自动创建
- 检验项目映射：将HIS系统的检验项目代码映射为LIS系统内部代码
- 数据校验：校验必填字段、检验项目有效性、标本类型合法性
- 事务处理：医嘱入库使用数据库事务，保证数据一致性
- 幂等性处理：根据消息控制ID（MSH-10）实现消息去重，防止重复处理

### 任务3：ORU^R01消息处理

#### 详细描述

实现ORU^R01（Observation Result Unsolicited）消息的构建与发送功能。ORU^R01消息由LIS系统发送，用于向HIS系统回传检验结果。处理流程包括：获取检验结果数据、构建ORU^R01消息、发送消息、处理ACK应答。

#### 输入

- **检验结果数据**：患者信息、就诊信息、检验医嘱信息、检验项目结果列表（项目代码、项目名称、结果值、单位、参考范围、异常标志、结果时间等）
- **发送配置**：目标HIS系统地址、端口号

#### 输出

- **ORU^R01消息**：构建完成的HL7消息文本
- **发送结果**：发送成功/失败状态、发送时间
- **ACK应答**：HIS系统返回的应答消息

#### 技术要点

- 使用HAPI的ORU_R01模型类构建消息
- 消息结构规范：MSH（消息头）+ PID（患者信息）+ PV1（就诊信息）+ ORC（医嘱信息）+ OBR（检验医嘱）+ OBX（检验结果，可多个）
- 异常标志处理：根据结果值与参考范围的比较自动设置异常标志（N-正常、H-偏高、L-偏低、A-异常）
- 多结果项支持：一个检验医嘱可包含多个OBX结果段
- 消息编码：使用指定的字符集编码（默认UTF-8）
- 发送状态追踪：记录消息发送状态，支持失败重发
- ACK超时处理：发送后等待ACK应答，超时时间可配置（默认30秒）

### 任务4：ACK消息生成

#### 详细描述

实现HL7 ACK（Acknowledgment）消息的自动生成功能。ACK消息用于对收到的HL7消息进行应答确认，告知发送方消息的处理结果。支持三种应答类型：AA（Application Accept-接受）、AE（Application Error-错误）、AR（Application Reject-拒绝）。

#### 输入

- **原始消息**：收到的HL7原始消息（用于提取MSH信息）
- **应答类型**：AA/AE/AR
- **错误信息**：错误代码、错误描述（AE/AR时必填）

#### 输出

- **ACK消息**：符合HL7标准的ACK消息文本
- **MSA段**：消息确认段（包含消息控制ID、应答代码、错误信息）

#### 技术要点

- ACK消息必须包含MSA（Message Acknowledgment）段和MSH（Message Header）段
- MSA-1设置应答代码：AA（接受）、AE（错误）、AR（拒绝）
- MSA-2设置原始消息的消息控制ID（MSH-10），用于消息关联
- 错误时应包含ERR段（Error Segment），详细描述错误原因
- ACK消息的MSH字段需正确设置：发送方与接收方互换、消息类型设为ACK
- 支持延迟ACK模式（先返回AA，异步处理后再发送最终结果）
- ACK消息生成时间精确到毫秒

### 任务5：MLLP协议通讯

#### 详细描述

实现MLLP（Minimum Lower Layer Protocol）协议通讯层，作为HL7消息的传输协议。MLLP基于TCP/IP，定义了消息的分帧规则（起始块、数据块、结束块），确保HL7消息在网络上可靠传输。

#### 输入

- **连接配置**：服务端模式（监听端口）/ 客户端模式（目标IP、目标端口）
- **发送数据**：HL7消息文本
- **接收数据**：TCP字节流

#### 输出

- **连接状态**：已连接/未连接/连接异常
- **发送结果**：发送成功/失败、发送字节数
- **接收结果**：完整的HL7消息文本

#### 技术要点

- **MLLP分帧规则**：
  - 起始块：0x0B（VT垂直制表符）
  - 数据块：HL7消息文本
  - 结束块：0x1C（FS文件分隔符）+ 0x0D（CR回车符）
- 使用Netty框架实现MLLP协议编解码器（MllpEncoder、MllpDecoder）
- **服务端模式**：监听指定端口，接收HIS系统发送的HL7消息
- **客户端模式**：主动连接HIS系统，发送ORU^R01消息
- 支持长连接模式，避免频繁建立/断开TCP连接
- 连接心跳检测：定时发送心跳包维持连接活跃状态
- 粘包/拆包处理：正确处理TCP流的粘包和拆包问题
- 支持TLS/SSL加密传输（可选配置）
- 连接池管理：支持多个HIS系统同时连接

### 任务6：消息日志记录

#### 详细描述

实现HL7消息的完整日志记录功能，记录所有发送和接收的HL7消息，支持消息状态追踪和问题排查。日志内容包括消息原始内容、消息类型、发送/接收时间、处理状态、错误信息等。

#### 输入

- **日志记录请求**：消息方向（发送/接收）、消息类型、消息内容、处理状态、关联业务ID
- **日志查询请求**：消息方向、消息类型、时间范围、处理状态、患者ID、分页参数

#### 输出

- **记录结果**：日志保存成功/失败
- **查询结果**：消息日志列表（分页）、总记录数、消息详情

#### 技术要点

- 消息日志表（hl7_message_log）记录完整的消息交互信息
- 日志字段包括：消息ID、消息控制ID（MSH-10）、消息方向（SEND/RECEIVE）、消息类型（ORM/ORU/ACK等）、消息内容（完整HL7文本）、处理状态（SUCCESS/FAIL/PROCESSING）、错误信息、发送方、接收方、创建时间
- 消息内容使用TEXT类型存储，保留完整的HL7消息原文
- 支持按消息方向、消息类型、处理状态、时间范围等多条件组合查询
- 支持通过消息控制ID（MSH-10）精确追踪消息的完整交互链路
- 日志数据保留策略：默认保留90天，超期数据自动归档
- 消息内容敏感信息脱敏（患者姓名、身份证号等）
- 提供日志导出功能（CSV/Excel格式）

### 任务7：接口配置管理

#### 详细描述

实现HL7接口的配置管理功能，支持配置多个HIS系统的连接参数和接口参数。配置信息包括连接参数（IP地址、端口号）、通讯参数（超时时间、重试次数、编码方式）、业务参数（消息版本、字段映射规则）等。

#### 输入

- **新增配置**：系统名称、系统编码、连接方式（客户端/服务端）、IP地址、端口号、超时时间、重试次数、编码方式、HL7版本、启用状态
- **修改配置**：配置ID及需要修改的字段
- **查询配置**：系统编码、启用状态

#### 输出

- **操作结果**：成功/失败状态

- 配置修改后实时生效（使用Redis缓存配置，修改时更新缓存）

### 任务8：异常处理与重试机制

#### 详细描述

建立完善的HL7消息处理异常处理与重试机制，确保消息处理的可靠性。包括消息解析异常处理、业务处理异常处理、网络通讯异常处理、消息重发机制等。

#### 输入

- **异常事件**：解析异常、业务异常、网络异常、超时异常
- **重试请求**：消息ID、重试次数

#### 输出

- **异常处理结果**：异常分类、异常描述、处理方式（重试/告警/人工处理）
- **重试结果**：重试成功/失败、重试次数、下次重试时间

#### 技术要点

- **异常分类处理**：
  - 解析异常：记录原始消息和错误详情，返回AE类型ACK，不重试
  - 业务异常：记录错误详情，返回AE类型ACK，不重试
  - 网络异常：自动重试，返回AR类型ACK
  - 超时异常：自动重试，记录超时日志
- **重试策略**：
  - 最大重试次数：3次（可配置）
  - 重试间隔：采用固定间隔（10秒/30秒/60秒）或指数退避算法
  - 重试队列：使用消息队列（RabbitMQ）实现异步重试
  - 死信队列：超过最大重试次数的消息进入死信队列，等待人工处理
- **告警机制**：
  - 连续失败超过阈值时触发告警（默认连续失败5次）
  - 告警方式：系统日志、邮件通知（可扩展短信通知）
  - 告警内容包括：异常类型、异常详情、影响范围、建议处理方式
- **熔断机制**：当某个HIS系统连续失败超过阈值时，自动熔断该连接，避免影响其他系统

## 五、技术约束

### 5.1 技术栈要求

| 技术 | 版本要求 | 用途 |
|------|----------|------|
| Java | JDK 17+ | 开发语言 |
| Spring Boot | 2.7.18 | 应用框架 |
| HAPI-HL7 | 2.5.x | HL7消息解析与构建 |
| Netty | 4.1.x | MLLP协议通讯 |
| MyBatis-Plus | 3.5.x | ORM框架 |
| MySQL | 8.0+ | 数据库 |
| RabbitMQ | 3.12.x | 消息队列（重试机制） |
| Redis | 7.0+ | 缓存（配置缓存） |
| Maven | 3.8+ | 构建工具 |

### 5.2 编码规范

- 遵循阿里巴巴Java开发手册编码规范
- HL7消息处理类命名以"Handler"结尾（如OrmO01Handler、OruR01Handler）
- 编解码器命名以"Encoder"/"Decoder"结尾（如MllpEncoder、MllpDecoder）
- 每个消息处理器需添加详细的处理流程注释
- 异常处理使用统一异常处理类，按异常类型分类处理
- 日志记录使用SLF4J + Logback，日志级别使用规范（DEBUG-调试、INFO-业务、WARN-告警、ERROR-异常）

### 5.3 接口规范

- HL7消息接口遵循HL7 v2.x标准
- MLLP通讯端口默认：8086
- 管理API采用RESTful风格：/api/hl7/*
- 统一响应格式：{code, message, data}
- 消息处理支持同步和异步两种模式

### 5.4 安全约束

- HL7通讯支持TLS/SSL加密传输
- 接口配置信息加密存储
- 消息日志中敏感信息脱敏处理
- 连接IP白名单限制
- 消息防篡改校验（可选）

## 【Git工作流规范】

> 本章节为V3.0新增，所有规则具有强制约束力。详细规范参见 `/workspace/lis-project/docs/05-开发规范/Git版本控制规范.md`。

### 分支管理

- **工作分支**：`feature/agent-06-hl7-service`
- **修复分支**：`bugfix/agent-06-{description}`
- **禁止**直接在`main`或`develop`分支上提交任何代码
- 分支由项目经理（PM）创建，你只能在分配的分支上工作

### 提交规范

- **格式**：`[Agent-06] {类型}({范围}): {简述}`
- **类型**：feat（新功能）、fix（修复）、refactor（重构）、docs（文档）、test（测试）、chore（构建/配置）
- **示例**：`[Agent-06] feat(HL7服务): 新增XXX功能`
- 每个功能点至少一次提交，禁止积压大量变更后一次性提交
- 提交前必须确保代码可编译、功能可运行

### 过程文档要求

每个功能点完成后，必须同步提交过程文档：

| 文档类型 | 存放路径 | 提交时机 |
|---------|---------|---------|
| 功能开发记录 | `/workspace/lis-project/docs/process/Agent-06/{功能名称}-开发记录.md` | 每个功能点完成时 |
| 变更日志 | `/workspace/lis-project/docs/process/Agent-06/CHANGELOG.md` | 每次提交时更新 |

过程文档模板参见：`/workspace/lis-project/docs/process/开发记录模板.md`

### 阶段交付要求

完成本阶段全部任务后，必须提交以下交付物（缺一不可）：

1. ✅ 阶段开发报告（按D09《开发报告模板》格式）
2. ✅ CHANGELOG变更日志
3. ✅ 所有功能点的开发记录（`docs/process/Agent-06/`目录下）
4. ✅ 数据库变更脚本（如有数据库结构变更，含回滚脚本）
5. ✅ 通知项目经理进行合并审查

### PM汇报机制

- **每完成一个功能点**：Git commit + 过程文档 + 通知PM（附commit hash）
- **遇到阻塞问题**：立即通知PM，不得自行跳过或绕过
- **每日结束前**：提交当日进度总结到CHANGELOG
- **合并请求**：全部完成后，等待PM审查代码和文档，经PM审批后方可合并到develop

### 违规后果

| 违规行为 | 后果 |
|---------|------|
| 直接在main/develop上提交 | PM执行git reset回退 + 警告 |
| 提交信息不规范 | PM驳回提交，要求修改 |
| 过程文档缺失 | PM驳回合并请求 |
| 未经审批自行合并 | PM回退合并 + 暂停Agent |
| 虚假代码/文档 | PM回退分支 + 按红线禁令R1处理 |
