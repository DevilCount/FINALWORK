# 实验室管理系统 - 编码规范

> 本文档为"基于微服务架构的实验室管理系统"项目的统一编码规范，适用于所有参与开发的AI子Agent及开发人员。项目采用前后端分离架构，后端基于 Java/Spring Boot，前端基于 Vue.js 3/TypeScript，9个AI子Agent并行开发各微服务模块。

---

## 一、后端编码规范（Java/Spring Boot）

### 1.1 项目结构规范

#### 1.1.1 包命名规范

所有包名统一使用小写字母，采用反向域名格式，基础包路径为 `com.lis`。各微服务模块的包结构如下：

```
com.lis.{模块名}
├── controller          # 控制层，负责接收和响应HTTP请求
├── service             # 业务逻辑层接口
│   └── impl            # 业务逻辑层实现
├── mapper              # 数据访问层（MyBatis-Plus Mapper接口）
├── entity              # 数据库实体类（DO）
├── dto                 # 数据传输对象
│   ├── request         # 请求DTO
│   └── response        # 响应DTO
├── vo                  # 视图对象
├── config              # 配置类
├── common              # 公共模块
│   ├── constant        # 常量定义
│   ├── enums           # 枚举类
│   ├── exception       # 自定义异常
│   ├── handler         # 全局处理器
│   └── util            # 工具类
├── converter           # 对象转换器（MapStruct）
└── aspect              # AOP切面
```

#### 1.1.2 类命名规范

| 层级 | 后缀 | 示例 |
|------|------|------|
| Controller | `Controller` | `UserController` |
| Service接口 | `Service` | `UserService` |
| Service实现 | `ServiceImpl` | `UserServiceImpl` |
| Mapper接口 | `Mapper` | `UserMapper` |
| 数据库实体 | `DO` 或无后缀 | `UserDO` 或 `User` |
| 请求DTO | `Request` 或 `ReqDTO` | `CreateUserRequest` |
| 响应DTO | `Response` 或 `RespDTO` | `UserResponse` |
| 视图对象 | `VO` | `UserVO` |
| 配置类 | `Config` 或 `Configuration` | `RedisConfig` |
| 工具类 | `Utils` 或 `Helper` | `DateUtils` |
| 常量类 | `Constants` | `CommonConstants` |
| 枚举类 | `Enum` | `OrderStatusEnum` |
| 异常类 | `Exception` | `BusinessException` |

#### 1.1.3 分层架构规范

系统严格遵循分层架构，各层之间不允许跨层调用：

- **Controller层**：仅负责参数接收、校验、调用Service、返回结果，禁止包含业务逻辑。
- **Service层**：负责核心业务逻辑处理，可调用Mapper和其他Service。
- **Mapper层**：仅负责数据库操作，禁止包含业务逻辑。
- **禁止规则**：Controller禁止直接调用Mapper；禁止反向依赖（如Service调用Controller）。

### 1.2 代码风格

#### 1.2.1 命名规范

**包名**：全小写，单词间不使用分隔符，如 `com.lis.usercenter`。

**类名**：使用 UpperCamelCase（大驼峰），名词形式，如 `OrderController`、`UserService`。

**方法名**：使用 lowerCamelCase（小驼峰），动词开头，如 `getUserById()`、`createOrder()`。

**常量名**：全大写，单词间用下划线分隔，如 `MAX_RETRY_COUNT`。

**变量名**：使用 lowerCamelCase，如 `userName`、`orderList`。布尔类型变量以 `is`/`has`/`can` 开头，如 `isActive`、`hasPermission`。

**通用方法命名约定**：

| 操作 | 方法名前缀 | 示例 |
|------|-----------|------|
| 查询单个 | `get` | `getUserById` |
| 查询列表 | `list` / `find` | `listUsersByDept` |
| 新增 | `save` / `add` / `create` | `createUser` |
| 修改 | `update` / `modify` | `updateUserStatus` |
| 删除 | `delete` / `remove` | `deleteUserById` |
| 判断 | `is` / `has` / `check` | `isUserExists` |
| 统计 | `count` | `countActiveUsers` |

#### 1.2.2 注释规范

**类注释**：每个类必须包含类级别的Javadoc注释，说明类的用途、作者和创建日期。

```java
/**
 * 用户管理服务实现类
 * <p>
 * 提供用户的增删改查、权限分配、状态管理等核心业务功能
 * </p>
 *
 * @author Agent-01
 * @since 2025-01-01
 */
@Service
public class UserServiceImpl implements UserService {
```

**方法注释**：所有public方法必须包含Javadoc注释，说明功能、参数和返回值。

```java
/**
 * 根据用户ID查询用户详细信息
 *
 * @param userId 用户唯一标识
 * @return 用户详情VO，若用户不存在则返回null
 * @throws BusinessException 当用户ID为空时抛出
 */
public UserVO getUserById(Long userId) {
```

**行内注释**：使用 `//` 进行行内注释，注释应放在代码上方或右侧，说明"为什么"而非"是什么"。

```java
// 使用Redis缓存减少数据库查询压力，缓存过期时间30分钟
String cacheKey = RedisKeyBuilder.buildUserKey(userId);
```

**TODO注释**：使用标准格式标记待办事项。

```java
// TODO Agent-03 2025-02-01: 需要增加批量导入功能的性能优化
```

#### 1.2.3 格式规范

- **缩进**：使用4个空格缩进，禁止使用Tab。
- **行宽**：每行不超过120个字符。
- **大括号**：左大括号不换行，右大括号独占一行（K&R风格）。
- **空行**：方法之间空一行，逻辑段落之间空一行。
- **导入顺序**：依次为JDK标准库、第三方库、项目内部类，各组之间空一行。使用IDE自动优化import，禁止使用通配符导入（`*`）。
- **数组声明**：使用 `String[] args` 而非 `String args[]`。
- **Lambda表达式**：参数与箭头之间不加空格，如 `(a, b) -> a + b`。

### 1.3 Controller层规范

#### 1.3.1 RESTful API设计

API路径统一使用小写字母，单词间用连字符 `-` 分隔，资源名使用复数形式：

```
GET    /api/v1/users              # 查询用户列表
GET    /api/v1/users/{id}         # 查询单个用户
POST   /api/v1/users              # 创建用户
PUT    /api/v1/users/{id}         # 更新用户（全量）
PATCH  /api/v1/users/{id}/status  # 更新用户状态（部分）
DELETE /api/v1/users/{id}         # 删除用户
```

API版本号通过URL路径管理，格式为 `/api/v{版本号}/{资源名}`。

#### 1.3.2 参数校验

使用 `@Valid` 或 `@Validated` 注解进行参数校验，校验规则通过注解定义在DTO上：

```java
@PostMapping
public Result<UserVO> createUser(@RequestBody @Valid CreateUserRequest request) {
    // ...
}

public class CreateUserRequest {
    @NotBlank(message = "用户名不能为空")
    @Length(min = 2, max = 20, message = "用户名长度为2-20个字符")
    private String username;

    @NotBlank(message = "手机号不能为空")
    @Pattern(regexp = "^1[3-9]\\d{9}$", message = "手机号格式不正确")
    private String phone;

    @Email(message = "邮箱格式不正确")
    private String email;
}
```

#### 1.3.3 统一响应格式

所有API响应统一使用 `Result<T>` 包装：

```java
@Data
public class Result<T> {
    private Integer code;       // 业务状态码
    private String message;     // 提示信息
    private T data;             // 响应数据
    private Long timestamp;     // 时间戳

    public static <T> Result<T> success(T data) {
        Result<T> result = new Result<>();
        result.setCode(200);
        result.setMessage("操作成功");
        result.setData(data);
        result.setTimestamp(System.currentTimeMillis());
        return result;
    }

    public static <T> Result<T> error(Integer code, String message) {
        Result<T> result = new Result<>();
        result.setCode(code);
        result.setMessage(message);
        result.setTimestamp(System.currentTimeMillis());
        return result;
    }
}
```

分页查询统一使用 `PageResult<T>`：

```java
@Data
public class PageResult<T> {
    private List<T> records;    // 数据列表
    private Long total;         // 总记录数
    private Long pageNum;       // 当前页码
    private Long pageSize;      // 每页大小
}
```

### 1.4 Service层规范

#### 1.4.1 事务管理

- 使用 `@Transactional` 注解声明事务，默认使用 `REQUIRED` 传播行为。
- `@Transactional` 注解添加在Service实现类的方法上，不添加在接口上。
- 只读操作使用 `@Transactional(readOnly = true)` 提升性能。
- 事务方法必须是 `public`，避免 `self-invocation` 导致事务失效。
- 事务边界尽量小，避免大事务。禁止在事务中调用外部API或进行文件操作。

```java
@Service
public class OrderServiceImpl implements OrderService {

    @Transactional(rollbackFor = Exception.class)
    @Override
    public void createOrder(CreateOrderRequest request) {
        // 1. 校验库存
        // 2. 创建订单
        // 3. 扣减库存
        // 4. 记录操作日志
    }

    @Transactional(readOnly = true)
    @Override
    public OrderVO getOrderDetail(Long orderId) {
        // 查询订单详情
    }
}
```

#### 1.4.2 异常处理

- 定义业务异常基类 `BusinessException`，通过错误码区分异常类型。
- Service层抛出 `BusinessException`，由全局异常处理器统一捕获处理。
- 禁止在Service层使用 `try-catch` 吞掉异常，异常必须向上传播或转换为业务异常。
- 禁止在Controller层处理业务异常。

```java
@Getter
public class BusinessException extends RuntimeException {
    private final Integer code;

    public BusinessException(String message) {
        super(message);
        this.code = 500;
    }

    public BusinessException(Integer code, String message) {
        super(message);
        this.code = code;
    }
}
```

全局异常处理器示例：

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(BusinessException.class)
    public Result<Void> handleBusinessException(BusinessException e) {
        log.warn("业务异常: {}", e.getMessage());
        return Result.error(e.getCode(), e.getMessage());
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public Result<Void> handleValidationException(MethodArgumentNotValidException e) {
        String message = e.getBindingResult().getFieldErrors().stream()
            .map(FieldError::getDefaultMessage)
            .collect(Collectors.joining(", "));
        return Result.error(400, message);
    }
}
```

#### 1.4.3 日志规范

- 使用 `@Slf4j`（Lombok）注解引入日志对象，禁止手动创建Logger。
- 日志级别使用规范：

| 级别 | 使用场景 |
|------|---------|
| `ERROR` | 系统异常、严重错误，需要立即关注 |
| `WARN` | 业务异常、非预期行为，但不影响主流程 |
| `INFO` | 关键业务节点、方法入口/出口（仅限核心方法） |
| `DEBUG` | 调试信息，开发环境使用，生产环境关闭 |

- 日志格式要求：使用占位符 `{}` 而非字符串拼接。

```java
// 正确
log.info("用户登录成功, userId={}, ip={}", userId, ip);

// 错误
log.info("用户登录成功, userId=" + userId + ", ip=" + ip);
```

- 异常日志必须打印完整堆栈：

```java
// 正确
log.error("处理订单失败, orderId={}", orderId, exception);

// 错误
log.error("处理订单失败: " + exception.getMessage());
```

### 1.5 Mapper层规范

#### 1.5.1 MyBatis-Plus使用规范

- 所有Mapper接口继承 `BaseMapper<T>`。
- 禁止在Service层直接使用 `BaseMapper` 的方法，必须通过自定义Mapper方法调用。
- 复杂查询使用XML映射文件，简单CRUD使用注解或Wrapper。
- 使用 `LambdaQueryWrapper` 替代 `QueryWrapper`，避免硬编码字段名。

```java
@Mapper
public interface UserMapper extends BaseMapper<UserDO> {

    /**
     * 查询用户详情（关联角色信息）
     */
    UserDetailVO selectUserDetailById(@Param("userId") Long userId);

    /**
     * 分页查询用户列表（多条件）
     */
    IPage<UserVO> selectUserPage(Page<UserVO> page, @Param("query") UserQueryRequest query);
}
```

#### 1.5.2 SQL编写规范

- SQL关键字统一大写：`SELECT`、`FROM`、`WHERE`、`LEFT JOIN` 等。
- 表名和字段名使用反引号包裹（MySQL），避免与保留字冲突。
- 禁止使用 `SELECT *`，必须明确列出所需字段。
- WHERE条件中禁止对索引列使用函数操作。
- 大数据量查询必须使用分页，禁止一次性查询全表。
- 使用 `#{}` 参数绑定，禁止使用 `${}`（防止SQL注入）。
- 多表JOIN不超过3张表，超过时考虑拆分查询或使用子查询。

```xml
<select id="selectUserPage" resultType="com.lis.usercenter.vo.UserVO">
    SELECT
        u.id,
        u.username,
        u.real_name,
        u.email,
        u.phone,
        d.name AS dept_name,
        r.name AS role_name
    FROM `sys_user` u
    LEFT JOIN `sys_dept` d ON u.dept_id = d.id
    LEFT JOIN `sys_user_role` ur ON u.id = ur.user_id
    LEFT JOIN `sys_role` r ON ur.role_id = r.id
    WHERE u.deleted = 0
    <if test="query.username != null and query.username != ''">
        AND u.username LIKE CONCAT('%', #{query.username}, '%')
    </if>
    <if test="query.deptId != null">
        AND u.dept_id = #{query.deptId}
    </if>
    ORDER BY u.create_time DESC
</select>
```

### 1.6 实体类规范

#### 1.6.1 DO/DTO/VO区分

| 类型 | 全称 | 用途 | 使用层级 |
|------|------|------|---------|
| DO | Data Object | 数据库实体映射 | Mapper层 |
| DTO | Data Transfer Object | 服务间数据传输 | Service层 |
| VO | View Object | 前端展示数据 | Controller层 |

**DO（数据库实体）**：

```java
@Data
@TableName("sys_user")
public class UserDO {
    @TableId(type = IdType.ASSIGN_ID)
    private Long id;

    private String username;

    private String password;

    private String realName;

    private String email;

    private String phone;

    private Integer status;

    @TableField(fill = FieldFill.INSERT)
    private LocalDateTime createTime;

    @TableField(fill = FieldFill.INSERT_UPDATE)
    private LocalDateTime updateTime;

    @TableLogic
    private Integer deleted;
}
```

**DTO（数据传输对象）**：

```java
@Data
public class CreateUserRequest {
    @NotBlank(message = "用户名不能为空")
    private String username;

    @NotBlank(message = "密码不能为空")
    @Length(min = 6, max = 20, message = "密码长度为6-20个字符")
    private String password;

    private String realName;
    private String email;
    private String phone;
    private Long deptId;
    private List<Long> roleIds;
}
```

**VO（视图对象）**：

```java
@Data
public class UserVO {
    private Long id;
    private String username;
    private String realName;
    private String email;
    private String phone;
    private String deptName;
    private String roleName;
    private Integer status;
    private String statusDesc;
    private LocalDateTime createTime;
}
```

使用 MapStruct 进行对象转换，禁止手动 `set` 属性：

```java
@Mapper(componentModel = "spring")
public interface UserConverter {
```

### 1.7 配置规范

#### 1.7.1 application.yml 规范

- 配置文件按环境分离：`application.yml`（公共配置）、`application-dev.yml`（开发环境）、`application-prod.yml`（生产环境）。
- 禁止在配置文件中硬编码密码、密钥等敏感信息，使用环境变量或配置中心管理。
- 配置项使用小写字母，单词间用连字符分隔，按功能模块分组。

```yaml
server:
  port: 8080
  servlet:
    context-path: /api/v1

spring:
  datasource:

    host: ${REDIS_HOST:localhost}
```

#### 1.7.2 Nacos配置规范

- 配置中心命名空间按环境隔离：`dev`、`test`、`prod`。
- 配置Group按服务模块划分：`USER_CENTER`、`DEVICE_MANAGE`、`REAGENT_MANAGE` 等。
- 配置Data ID格式：`{服务名}-{环境}.yml`，如 `lis-user-center-dev.yml`。
- 敏感配置使用Nacos的配置加密功能。

### 1.8 Git提交规范

#### 1.8.1 Commit Message格式

采用 Angular 规范，格式如下：

```
<type>(<scope>): <subject>

<body>

<footer>
```

**type（类型）**：

| 类型 | 说明 |
|------|------|
| `feat` | 新功能 |
| `fix` | 修复Bug |
| `docs` | 文档变更 |
| `style` | 代码格式调整（不影响逻辑） |
| `refactor` | 重构（既非新功能也非Bug修复） |
| `perf` | 性能优化 |
| `test` | 测试相关 |
| `chore` | 构建过程或辅助工具变动 |

**scope（范围）**：模块名称，如 `user`、`device`、`reagent`。

**subject（主题）**：简短描述，不超过50个字符，使用祈使句。

**示例**：

```
feat(user): 新增用户批量导入功能

- 支持Excel文件上传解析
- 支持数据校验和错误提示
- 支持异步导入，避免大文件阻塞

Agent-01
```

#### 1.8.2 分支策略

采用 Git Flow 简化版：

| 分支 | 命名规则 | 说明 |
|------|---------|------|
| `main` | `main` | 主分支，生产环境代码 |
| `develop` | `develop` | 开发分支，集成各模块最新代码 |
| `feature` | `feature/{模块名}-{功能描述}` | 功能分支，如 `feature/user-batch-import` |
| `hotfix` | `hotfix/{问题描述}` | 紧急修复分支 |
| `release` | `release/v{版本号}` | 发布分支 |

**分支操作规则**：
- 各Agent在自己的 `feature` 分支上开发，完成后提交Merge Request到 `develop`。
- 禁止直接向 `main` 分支推送代码。
- 合并前必须通过代码审查（Code Review）和自动化测试。
- 功能分支命名需包含Agent编号前缀，如 `feature/agent01-user-export`。

### 2.4 状态管理规范（Pinia）

#### 2.4.1 Store定义

使用 Setup Store 风格定义Store，按模块拆分：

```typescript
// stores/modules/user.ts
import { defineStore } from 'pinia'
import { ref, computed } from 'vue'
import { login, getUserInfo, logout } from '@/api/modules/user'
import type { LoginParams, UserInfo } from '@/types/api'

export const useUserStore = defineStore('user', () => {
  // State
  const token = ref<string>(localStorage.getItem('token') || '')
  const userInfo = ref<UserInfo | null>(null)
  const permissions = ref<string[]>([])

  // Getters
  const isLoggedIn = computed(() => !!token.value)
  const username = computed(() => userInfo.value?.username || '')

  // Actions
  async function loginAction(params: LoginParams) {
    const res = await login(params)
    token.value = res.data.token
    localStorage.setItem('token', res.data.token)
  }

  async function getUserInfoAction() {
    const res = await getUserInfo()
    userInfo.value = res.data
    permissions.value = res.data.permissions || []
  }

  function logoutAction() {
    token.value = ''
    userInfo.value = null
    permissions.value = []
    localStorage.removeItem('token')
  }

  function hasPermission(permission: string): boolean {
    return permissions.value.includes(permission)
  }

  return {
    token,
    userInfo,
    permissions,
    isLoggedIn,
    username,
    loginAction,
    getUserInfoAction,
    logoutAction,
    hasPermission,
  }
})
```

### 2.5 API调用规范

#### 2.5.1 Axios封装

统一封装Axios实例，处理请求拦截、响应拦截、错误处理：

```typescript
// api/request.ts
import axios from 'axios'
import type { AxiosInstance, AxiosRequestConfig, AxiosResponse } from 'axios'
import { useUserStore } from '@/stores/modules/user'
import { ElMessage } from 'element-plus'

const service: AxiosInstance = axios.create({
  baseURL: import.meta.env.VITE_API_BASE_URL,
  timeout: 30000,
  headers: { 'Content-Type': 'application/json' },
})

// 请求拦截器
service.interceptors.request.use(
  (config) => {
    const userStore = useUserStore()
    if (userStore.token) {
      config.headers.Authorization = `Bearer ${userStore.token}`
    }
    return config
  },
  (error) => Promise.reject(error)
)

// 响应拦截器
service.interceptors.response.use(
  (response: AxiosResponse) => {
    const { code, message, data } = response.data
    if (code === 200) {
      return data
    }
    ElMessage.error(message || '请求失败')
    return Promise.reject(new Error(message))
  },
  (error) => {
    if (error.response?.status === 401) {
      const userStore = useUserStore()
      userStore.logoutAction()
      window.location.href = '/login'
    }
    ElMessage.error(error.response?.data?.message || '网络异常')
    return Promise.reject(error)
  }
)

export default service
```

## 三、数据库规范

### 3.1 表设计规范

#### 3.1.1 表命名规范

- 表名使用小写字母，单词间用下划线 `_` 分隔。
- 每个微服务的表使用统一前缀标识，如 `sys_`（系统管理）、`dev_`（设备管理）、`reagent_`（试剂管理）、`exp_`（实验管理）。
- 关联表使用 `_{主表}_{从表}` 格式命名，如 `sys_user_role`。
- 禁止使用数据库保留字作为表名。

#### 3.1.2 字段规范

- 字段名使用小写字母，单词间用下划线 `_` 分隔。
- 每个表必须包含以下公共字段：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `id` | `BIGINT` | 主键，使用雪花算法生成 |
| `create_time` | `DATETIME` | 创建时间 |
| `create_by` | `BIGINT` | 创建人ID |
| `update_time` | `DATETIME` | 更新时间 |
| `update_by` | `BIGINT` | 更新人ID |
| `deleted` | `TINYINT` | 逻辑删除标志（0-未删除，1-已删除） |
| `remark` | `VARCHAR(500)` | 备注 |

- 字段类型选择原则：
  - 金额使用 `DECIMAL(10,2)`，禁止使用 `FLOAT` 或 `DOUBLE`。
  - 时间使用 `DATETIME`，禁止使用 `TIMESTAMP`（避免2038年问题）。
  - 字符串根据实际长度选择 `VARCHAR`，禁止滥用 `TEXT`。
  - 布尔值使用 `TINYINT(1)`，0表示false，1表示true。
  - 状态字段使用 `TINYINT` 配合注释说明各值含义。

- 所有字段必须添加 `NOT NULL` 约束（主键和逻辑删除字段除外），设置合理的默认值。
- 所有字段必须添加 `COMMENT` 注释。

#### 3.1.3 索引规范

- 主键使用聚簇索引，统一使用 `id` 字段。
- 唯一约束使用唯一索引。
- 高频查询条件字段创建普通索引。
- 联合索引遵循最左前缀原则，将区分度高的字段放在前面。
- 单表索引数量不超过5个。
- 禁止在低区分度字段（如性别、状态）上单独创建索引。
- 索引命名规范：`idx_{表名缩写}_{字段名}`，唯一索引：`uk_{表名缩写}_{字段名}`。

```sql
-- 创建索引示例
CREATE INDEX idx_user_dept ON sys_user(dept_id);
CREATE INDEX idx_user_phone ON sys_user(phone);
CREATE UNIQUE INDEX uk_user_username ON sys_user(username);
CREATE INDEX idx_order_user_status ON exp_order(user_id, status);
```

### 3.2 SQL编写规范

- SQL关键字统一大写。
- 表名和字段名使用反引号包裹。
- 禁止使用 `SELECT *`，必须明确列出所需字段。
- WHERE条件中禁止对索引列使用函数或隐式类型转换。
- 大数据量操作必须分批执行，单次操作不超过1000条。
- 禁止使用存储过程、触发器、视图（业务逻辑放在应用层）。
- DELETE操作统一使用逻辑删除（更新 `deleted` 字段）。
- 数据变更SQL必须先在测试环境验证，禁止直接在生产环境执行。

### 3.3 数据迁移规范

- 使用 Flyway 管理数据库版本变更。
- 迁移脚本命名格式：`V{版本号}__{描述}.sql`，如 `V1.0.1__add_user_table.sql`。
- 每个迁移脚本必须是幂等的（可重复执行不报错）。
- 禁止修改已执行的迁移脚本。
- 数据迁移脚本与结构迁移脚本分开管理。
- 重大数据变更必须编写回滚脚本：`U{版本号}__{描述}.sql`。

## 四、Docker部署规范

### 4.1 Dockerfile编写规范

- 基础镜像使用官方镜像，指定明确版本号，禁止使用 `latest` 标签。
- 使用多阶段构建（multi-stage build）减小镜像体积。
- 镜像层数尽量少，合并RUN指令。
- 禁止在镜像中存储密码、密钥等敏感信息。

- 设置合理的工作目录。

```yaml
    networks:
      - lis-network

networks:
  lis-network:
    driver: bridge

  mysql-data:
  redis-data:
```

### 4.3 环境变量管理

- 敏感信息（密码、密钥、Token）通过环境变量注入，禁止硬编码。
- 使用 `.env` 文件管理开发环境变量，`.env` 文件必须添加到 `.gitignore`。
- 提供 `.env.example` 文件作为模板，包含所有需要的环境变量及其说明。
- 环境变量命名使用大写字母和下划线，格式为 `{服务名}_{变量名}`。

```.env.example
# 数据库配置
DB_HOST=localhost
DB_PORT=3306
DB_NAME=lis_user
DB_USERNAME=root
DB_PASSWORD=your_password_here

# Redis配置
REDIS_HOST=localhost
REDIS_PORT=6379
REDIS_PASSWORD=your_redis_password_here

# Nacos配置
NACOS_HOST=localhost
NACOS_PORT=8848

# JWT配置
JWT_SECRET=your_jwt_secret_here
JWT_EXPIRATION=86400
```

## 五、通用规范

### 5.1 代码审查规范

- 所有代码合并到 `develop` 分支前必须通过Code Review。
- 审查重点：代码规范性、业务逻辑正确性、安全性、性能。
- 审查意见分为 `must-fix`（必须修改）、`suggestion`（建议修改）、`nitpick`（可选修改）。
- 每个Agent提交的代码至少需要1个其他Agent审查通过。

### 5.2 安全编码规范

- 所有用户输入必须进行校验和转义，防止XSS攻击。
- SQL参数必须使用预编译语句，防止SQL注入。
- 文件上传必须校验文件类型、大小和内容。
- 敏感数据（密码、身份证号等）必须加密存储和传输。
- API接口必须进行身份认证和权限校验。
- 禁止在日志中输出敏感信息。

### 5.3 性能编码规范

- 禁止在循环中执行数据库查询，使用批量操作。
- 大数据量查询必须使用分页。
- 频繁访问的数据使用缓存（Redis）。
- 异步操作使用消息队列（RabbitMQ/Kafka）。
- 前端列表页面必须使用虚拟滚动或分页加载。
