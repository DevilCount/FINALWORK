## 二、任务背景

### 2.1 业务背景

AI辅助诊断是LIS系统的智能化核心模块，旨在通过规则引擎对检验结果进行自动分析，辅助检验医师进行结果审核和临床判断。AI辅助诊断可以显著提高检验结果审核效率，减少人工疏漏，为临床提供更有价值的诊断参考建议。系统支持血常规、尿常规、肝功能等多个检验专业的智能分析，通过可配置的诊断规则实现灵活的诊断逻辑管理。

### 2.2 技术背景

AI辅助诊断服务采用基于规则引擎的实现方案。规则引擎负责加载、解析和执行诊断规则，根据检验结果数据与规则条件进行匹配，输出诊断结论和置信度。规则引擎可选择使用Drools开源规则引擎或自研轻量级规则引擎实现。诊断规则采用条件表达式定义，支持多条件组合、数值范围判断、逻辑运算等。

### 2.3 核心目标

1. 设计并实现灵活可扩展的规则引擎，支持规则的动态加载和执行
2. 实现血常规、尿常规、肝功能等常见检验项目的智能诊断分析
3. 建立科学的诊断结果置信度计算算法
4. 实现诊断记录的完整管理（保存、查询、审核）
5. 提供诊断规则的CRUD管理和启用/停用控制
6. 建立医师审核机制，确保AI诊断结果的安全可靠性

## 三、任务清单

### 任务1：规则引擎设计

#### 详细描述

设计并实现AI辅助诊断的核心规则引擎，负责诊断规则的加载、条件解析和规则执行。规则引擎是整个AI诊断服务的基础组件，需要具备良好的扩展性、高性能和可靠性。

#### 输入

- **规则定义**：规则ID、规则名称、规则类型（血常规/尿常规/肝功能等）、条件表达式（JSON格式）、优先级、启用状态
- **执行请求**：检验结果数据（项目代码、项目名称、结果值、单位、参考范围等）、规则类型

#### 输出

- **规则加载结果**：加载成功/失败、加载规则数量
- **规则执行结果**：匹配的规则列表、诊断结论、置信度评分
- **规则解析结果**：解析后的规则条件树结构

#### 技术要点

- **规则引擎选型**：评估Drools规则引擎与自研轻量级引擎的优劣，根据实际需求选择
  - Drools方案：功能强大，支持复杂规则，但引入依赖较重，学习成本高
  - 自研方案：轻量灵活，完全可控，适合当前业务场景，但需自行实现核心逻辑
- **规则条件表达式设计**：
  - 支持比较运算符：>、>=、<、<=、==、!=
  - 支持逻辑运算符：AND、OR、NOT
  - 支持数值范围判断：BETWEEN min AND max
  - 支持多条件组合：嵌套括号分组
- **规则加载机制**：
  - 应用启动时从数据库加载所有启用状态的规则
  - 规则变更时支持热加载（无需重启服务）
  - 使用Redis缓存规则数据，减少数据库查询
- **规则执行流程**：
  1. 接收检验结果数据
  2. 根据检验类型筛选适用的规则集
  3. 按规则优先级排序
  4. 逐条执行规则匹配
  5. 收集所有匹配的规则结论
  6. 计算综合置信度
  7. 返回诊断结果
- **规则条件解析器**：将JSON格式的条件表达式解析为可执行的条件树
- **性能优化**：规则预编译、条件索引、并行执行

### 任务2：血常规诊断

#### 详细描述

实现血常规检验项目的AI辅助诊断功能，根据白细胞（WBC）、红细胞（RBC）、血红蛋白（HGB）、血小板（PLT）、红细胞压积（HCT）、平均红细胞体积（MCV）、平均红细胞血红蛋白含量（MCH）、平均红细胞血红蛋白浓度（MCHC）、淋巴细胞百分比（LYM%）、中性粒细胞百分比（NEUT%）、单核细胞百分比（MONO%）、嗜酸性粒细胞百分比（EO%）等指标进行综合分析判断。

#### 输入

- **血常规检验结果**：WBC、RBC、HGB、PLT、HCT、MCV、MCH、MCHC、LYM%、NEUT%、MONO%、EO%等指标值及对应参考范围
- **患者信息**：性别、年龄（部分参考范围与性别年龄相关）

#### 输出

- **诊断结论**：可能的疾病或异常情况列表，如：
  - 白细胞增多：提示感染、炎症、白血病等可能
  - 白细胞减少：提示病毒感染、放射损伤、再生障碍性贫血等可能
  - 贫血判断：根据RBC、HGB、HCT综合判断贫血类型（小细胞低色素性贫血、大细胞性贫血、正细胞性贫血）
  - 血小板异常：血小板增多或减少的可能原因
  - 白细胞分类异常：淋巴细胞比例异常、中性粒细胞比例异常等
- **置信度**：每个诊断结论的置信度评分（0-100%）
- **建议措施**：建议进一步检查项目或临床关注事项

#### 技术要点

- 诊断规则需区分成人参考范围和儿童参考范围（根据年龄自动切换）
- 部分指标参考范围需区分性别（如RBC、HGB）
- 贫血分类判断逻辑：根据MCV、MCH、MCHC三个指标综合判断贫血类型
  - MCV降低 + MCH降低 + MCHC降低 → 小细胞低色素性贫血（提示缺铁性贫血）
  - MCV升高 + MCH升高 → 大细胞性贫血（提示巨幼细胞贫血）
  - MCV正常 + MCH正常 + MCHC正常 → 正细胞性贫血（提示急性失血、溶血性贫血）
- 多指标联合分析：当多个指标同时异常时，综合判断给出更精确的诊断建议
- 诊断结论需标注"仅供参考，以临床诊断为准"的免责声明

### 任务3：尿常规诊断

#### 详细描述

实现尿常规检验项目的AI辅助诊断功能，根据尿蛋白（PRO）、尿糖（GLU）、隐血（BLD）、白细胞（LEU）、亚硝酸盐（NIT）、尿胆原（URO）、胆红素（BIL）、酮体（KET）、比重（SG）、酸碱度（pH）、红细胞（ERY）等指标进行综合分析判断。

#### 输入

- **尿常规检验结果**：PRO、GLU、BLD、LEU、NIT、URO、BIL、KET、SG、pH、ERY等指标值及对应参考范围
- **患者信息**：性别、年龄

#### 输出

- **诊断结论**：可能的疾病或异常情况列表，如：
  - 尿蛋白阳性：提示肾炎、肾病综合征、糖尿病肾病等可能
  - 尿糖阳性：提示糖尿病、肾性糖尿等可能
  - 隐血阳性：提示泌尿系统结石、感染、肿瘤等可能
  - 白细胞阳性：提示泌尿系统感染可能
  - 亚硝酸盐阳性：提示细菌感染可能
  - 胆红素阳性：提示肝胆疾病可能
  - 酮体阳性：提示糖尿病酮症酸中毒、饥饿性酮症等可能
- **置信度**：每个诊断结论的置信度评分
- **建议措施**：建议进一步检查项目

#### 技术要点

- 尿常规指标多为定性或半定量结果（-、+/-、+1、+2、+3），需正确处理等级判断
- 多指标联合分析：
  - PRO + BLD + LEU同时阳性 → 高度提示泌尿系统感染
  - GLU + KET同时阳性 → 提示糖尿病酮症酸中毒风险
  - BIL + URO同时异常 → 提示肝胆系统疾病
- 尿液颜色与透明度可作为辅助判断依据
- 结合患者性别给出差异化的诊断建议（如女性患者泌尿系统感染概率更高）

### 任务4：肝功能诊断

#### 详细描述

实现肝功能检验项目的AI辅助诊断功能，根据谷丙转氨酶（ALT）、谷草转氨酶（AST）、总胆红素（TBIL）、直接胆红素（DBIL）、间接胆红素（IBIL）、总蛋白（TP）、白蛋白（ALB）、球蛋白（GLB）、白球比（A/G）、碱性磷酸酶（ALP）、谷氨酰转肽酶（GGT）等指标进行综合分析判断。

#### 输入

- **肝功能检验结果**：ALT、AST、TBIL、DBIL、IBIL、TP、ALB、GLB、A/G、ALP、GGT等指标值及对应参考范围
- **患者信息**：性别、年龄

#### 输出

- **诊断结论**：可能的疾病或异常情况列表，如：
  - 肝细胞损伤：ALT、AST升高，提示急性肝炎、慢性肝炎活动期、药物性肝损伤等
  - 胆道梗阻：ALP、GGT、DBIL升高，提示胆道结石、胆道肿瘤等
  - 肝功能衰竭：TP、ALB降低，A/G倒置，提示肝硬化、慢性肝病晚期
  - 黄疸类型判断：根据TBIL、DBIL、IBIL的比例判断肝细胞性黄疸、梗阻性黄疸或溶血性黄疸
  - 肝脏合成功能：TP、ALB水平评估肝脏合成功能
- **置信度**：每个诊断结论的置信度评分
- **建议措施**：建议进一步检查项目（如肝炎病毒标志物、肝脏影像学检查等）

#### 技术要点

- ALT/AST比值（De Ritis比值）分析：
  - AST/ALT > 1 → 提示酒精性肝病、肝硬化
  - AST/ALT > 2 → 高度提示酒精性肝病
  - AST/ALT < 1 → 提示急性病毒性肝炎
- 黄疸类型判断逻辑：
  - DBIL/TBIL > 20% → 梗阻性黄疸
  - DBIL/TBIL < 20% → 溶血性黄疸
  - DBIL/TBIL 在20%-50%之间 → 肝细胞性黄疸
- 肝功能损害程度分级：
  - 轻度：ALT/AST < 正常值3倍
  - 中度：ALT/AST 在正常值3-10倍
  - 重度：ALT/AST > 正常值10倍
- Child-Pugh评分相关指标纳入分析（ALB、TBIL等）
- 多指标联合分析提高诊断准确性

### 任务5：诊断结果置信度计算

#### 详细描述

设计并实现诊断结果置信度计算算法，为每个AI诊断结论计算合理的置信度评分。置信度反映AI诊断结论的可靠程度，帮助医师判断诊断建议的可信水平。采用多因子加权算法综合计算置信度。

#### 输入

- **诊断匹配信息**：匹配的规则数量、规则优先级、条件匹配程度
- **检验数据质量**：数据完整性、数据异常程度（偏离参考范围的程度）
- **规则质量**：规则的历史准确率、规则适用范围

#### 输出

- **置信度评分**：0-100%的置信度值
- **置信度等级**：高（>=80%）、中（50%-80%）、低（<50%）
- **置信度因子明细**：各影响因子的权重和得分

#### 技术要点

- **多因子加权算法**：
  ```
  置信度 = W1 * 条件匹配度 + W2 * 数据偏离度 + W3 * 规则覆盖度 + W4 * 规则历史准确率
  ```
  其中W1、W2、W3、W4为权重系数（W1+W2+W3+W4=1）
- **条件匹配度**（权重40%）：匹配的条件数量占规则总条件数量的比例
- **数据偏离度**（权重25%）：检验结果偏离参考范围的程度，偏离越大置信度越高
- **规则覆盖度**（权重20%）：匹配的规则数量占该类型规则总数的比例
- **规则历史准确率**（权重15%）：规则在历史诊断中被医师确认准确的比例
- 置信度阈值设置：
  - 高置信度（>=80%）：优先展示，标记为"高度疑似"
  - 中置信度（50%-80%）：正常展示，标记为"可能"
  - 低置信度（<50%）：降级展示，标记为"待排除"
- 置信度计算结果四舍五入保留1位小数
- 权重系数支持通过配置调整

### 任务6：诊断记录管理

#### 详细描述

实现AI诊断记录的完整管理功能，包括诊断记录的保存、查询、详情查看等操作。诊断记录保存每次AI辅助诊断的完整信息，包括输入的检验数据、诊断结论、置信度、诊断时间等。

#### 输入

- **保存诊断记录**：检验结果数据、诊断结论列表（结论内容、置信度、规则ID）、患者信息、标本信息、诊断时间
- **查询诊断记录**：患者ID、时间范围、诊断类型、审核状态、分页参数
- **诊断记录详情**：诊断记录ID

#### 输出

- **保存结果**：操作成功/失败状态、诊断记录ID
- **查询结果**：诊断记录列表（分页）、总记录数
- **详情结果**：诊断记录完整信息（输入数据、诊断结论、置信度、审核信息等）

#### 技术要点

- 诊断记录使用JSON格式存储诊断结论列表，支持灵活的结论数量
- 诊断记录关联患者信息和标本信息，支持多维度查询
- 诊断记录一旦保存不可修改（保证数据完整性和可追溯性）
- 支持按诊断类型（血常规/尿常规/肝功能）分类查询
- 支持按审核状态（待审核/已审核/已驳回）筛选查询
- 诊断记录查询支持关联查询患者姓名、标本编号等关联信息
- 诊断记录保留期限：默认保留5年

### 任务7：诊断规则管理

#### 详细描述

实现诊断规则的CRUD管理功能，支持诊断规则的新增、修改、删除、查询以及启用/停用操作。诊断规则是AI辅助诊断的核心知识库，规则管理功能为系统管理员提供便捷的规则维护能力。

#### 输入

- **新增规则**：规则名称、规则类型（血常规/尿常规/肝功能）、条件表达式（JSON格式）、诊断结论、诊断建议、优先级、启用状态
- **修改规则**：规则ID及需要修改的字段
- **删除规则**：规则ID（逻辑删除）
- **查询规则**：规则类型、启用状态、规则名称关键词、分页参数
- **启用/停用规则**：规则ID、目标状态

#### 输出

- **操作结果**：成功/失败状态
- **查询结果**：规则列表（分页）、总记录数
- **规则详情**：规则完整信息（含条件表达式解析后的可读格式）

#### 技术要点

- 条件表达式使用JSON格式定义，结构示例：
  ```json
  {
    "logic": "AND",
    "conditions": [
      {"field": "WBC", "operator": ">", "value": 10},
      {"field": "NEUT%", "operator": ">", "value": 75}
    ],
    "conclusion": "提示细菌感染可能",
    "suggestion": "建议结合临床体征和CRP检查综合判断"
  }
  ```
- 支持嵌套条件表达式（conditions中可嵌套子条件组）
- 规则优先级使用整数表示，数值越小优先级越高
- 规则启用/停用后实时生效（更新缓存）
- 规则修改后自动进行语法校验，校验失败不允许保存
- 提供规则测试功能：输入检验数据，模拟执行规则，查看匹配结果
- 规则变更需记录操作审计日志

### 任务8：诊断审核机制

#### 详细描述

建立AI诊断结果的医师审核机制，确保AI辅助诊断结论经过专业医师审核后方可生效。审核机制包括：审核待办列表、审核操作（通过/驳回）、审核意见填写、审核记录查询等功能。

#### 输入

- **审核请求**：诊断记录ID、审核结果（通过/驳回）、审核意见（驳回时必填）、审核人ID
- **审核待办查询**：审核人ID、分页参数
- **审核记录查询**：诊断记录ID、审核人ID、时间范围、审核结果

#### 输出

- **审核结果**：操作成功/失败状态
- **待办列表**：待审核的诊断记录列表
- **审核记录**：审核历史记录列表

#### 技术要点

- 审核流程：AI诊断 → 待审核 → 医师审核 → 通过/驳回
- 审核通过：诊断结论标记为"已确认"，可展示给临床
- 审核驳回：诊断结论标记为"已驳回"，需记录驳回原因，AI规则引擎可据此优化规则
- 审核超时提醒：诊断记录超过24小时未审核时，发送提醒通知
- 审核权限控制：仅具有审核权限的医师可执行审核操作
- 审核记录不可修改（保证审核过程的完整性和可追溯性）
- 审核驳回数据反馈至规则优化：统计驳回率较高的规则，提示规则维护人员优化
- 支持批量审核操作（提高审核效率）

## 五、技术约束

### 5.1 技术栈要求

| 技术 | 版本要求 | 用途 |
|------|----------|------|
| Java | JDK 17+ | 开发语言 |
| Spring Boot | 2.7.18 | 应用框架 |
| MyBatis-Plus | 3.5.x | ORM框架 |
| MySQL | 8.0+ | 数据库 |
| Redis | 7.0+ | 缓存（规则缓存） |
| Drools（可选） | 8.44.x | 规则引擎 |
| Maven | 3.8+ | 构建工具 |

### 5.2 编码规范

- 遵循阿里巴巴Java开发手册编码规范
- 规则引擎核心类命名以"RuleEngine"结尾
- 诊断处理器命名以"DiagnosisHandler"结尾（如BloodRoutineDiagnosisHandler）
- 条件解析器命名以"ConditionParser"结尾
- 置信度计算器命名以"ConfidenceCalculator"结尾
- 诊断规则JSON格式需添加JSON Schema校验
- 所有诊断结论必须包含免责声明

### 5.3 接口规范

- RESTful API设计风格
- 统一响应格式：{code, message, data}
- 接口版本号：/api/ai
- 诊断接口支持同步和异步两种调用模式
- 异步诊断通过消息队列实现，结果通过回调通知

### 5.4 安全约束

- AI诊断结果必须经过医师审核后方可生效
- 诊断规则修改需具有管理员权限
- 患者数据脱敏处理（诊断记录中不存储患者姓名、身份证号等敏感信息）
- 诊断接口需进行身份认证和权限校验
- 诊断日志完整记录，支持审计追溯
- AI诊断结论明确标注"仅供参考"

## 【Git工作流规范】

> 本章节为V3.0新增，所有规则具有强制约束力。详细规范参见 `/workspace/lis-project/docs/05-开发规范/Git版本控制规范.md`。

### 分支管理

- **工作分支**：`feature/agent-08-ai-service`
- **修复分支**：`bugfix/agent-08-{description}`
- **禁止**直接在`main`或`develop`分支上提交任何代码
- 分支由项目经理（PM）创建，你只能在分配的分支上工作

### 提交规范

- **格式**：`[Agent-08] {类型}({范围}): {简述}`
- **类型**：feat（新功能）、fix（修复）、refactor（重构）、docs（文档）、test（测试）、chore（构建/配置）
- **示例**：`[Agent-08] feat(AI服务): 新增XXX功能`
- 每个功能点至少一次提交，禁止积压大量变更后一次性提交
- 提交前必须确保代码可编译、功能可运行

### 过程文档要求

每个功能点完成后，必须同步提交过程文档：

| 文档类型 | 存放路径 | 提交时机 |
|---------|---------|---------|
| 功能开发记录 | `/workspace/lis-project/docs/process/Agent-08/{功能名称}-开发记录.md` | 每个功能点完成时 |
| 变更日志 | `/workspace/lis-project/docs/process/Agent-08/CHANGELOG.md` | 每次提交时更新 |

过程文档模板参见：`/workspace/lis-project/docs/process/开发记录模板.md`

### 阶段交付要求

完成本阶段全部任务后，必须提交以下交付物（缺一不可）：

1. ✅ 阶段开发报告（按D09《开发报告模板》格式）
2. ✅ CHANGELOG变更日志
3. ✅ 所有功能点的开发记录（`docs/process/Agent-08/`目录下）
4. ✅ 数据库变更脚本（如有数据库结构变更，含回滚脚本）
5. ✅ 通知项目经理进行合并审查

### PM汇报机制

- **每完成一个功能点**：Git commit + 过程文档 + 通知PM（附commit hash）
- **遇到阻塞问题**：立即通知PM，不得自行跳过或绕过
- **每日结束前**：提交当日进度总结到CHANGELOG
- **合并请求**：全部完成后，等待PM审查代码和文档，经PM审批后方可合并到develop

### 违规后果

| 违规行为 | 后果 |
|---------|------|
| 直接在main/develop上提交 | PM执行git reset回退 + 警告 |
| 提交信息不规范 | PM驳回提交，要求修改 |
| 过程文档缺失 | PM驳回合并请求 |
| 未经审批自行合并 | PM回退合并 + 暂停Agent |
| 虚假代码/文档 | PM回退分支 + 按红线禁令R1处理 |
